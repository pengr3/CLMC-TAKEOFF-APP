<decisions>
## Implementation Decisions

### Phase A ‚Äî Scale Ratio Input (GAP-T2-00)

- **D-01:** Ratio input lives on a **new "Type ratio" tab** inside the existing `CalibrationDialog.tsx`, alongside the current "Draw line" tab. Keeps all calibration in one place; user picks the mode before touching the canvas.
- **D-02:** Input UX is **two side-by-side numeric fields** separated by a colon ‚Äî `[ 1 ] : [ 100 ]`. The left field defaults to `1` (locked). The estimator types the denominator into the right field. No text parsing, no ambiguity. Example: type `100` into the right box ‚Üí ratio is 1:100.
- **D-03:** Physical page size is **auto-derived from PDF.js `page.view`** (`[0, 0, widthPt, heightPt]` at 72 pts/inch ‚Üí mm via `√ó 25.4 / 72`). The derived sheet size (e.g. "841 √ó 594 mm ‚Äî A1") is shown to the user for confirmation before the calibration is accepted. If the derived size looks wrong (scanned PDF with bad metadata), user can fall back to the Draw Line tab.

### Phase B ‚Äî Markup Geometry Editing (GAP-T1-01 Vertex Edit + GAP-T1-02 Translate)

*Bundled into one phase ‚Äî they share the same Konva drag + stage inverse-transform infrastructure and require a single research pass.*

- **D-04:** Vertex edit mode activates by **clicking to select, then clicking again** on the already-selected markup. First click ‚Üí selection ring (existing behavior). Second click on the selected shape ‚Üí vertex edit mode with handles visible.
- **D-05:** Vertex handles are **small filled squares ‚Äî 8 √ó 8px, white fill, colored border** matching the markup color. CAD-style convention; any estimator who has used AutoCAD or Bluebeam will recognize it immediately.
- **D-06:** Exit vertex edit mode with **Enter to commit OR click outside the markup to commit** (when not mid-drag on a handle). Escape restores original vertex positions and exits without saving. The "click outside" path is guarded: if the user is mid-drag on a handle, releasing the mouse ends the drag but does not commit/exit.
- **D-07:** Markup translate triggers by **dragging a selected markup** with LMB (same 4px movement threshold as rubber-band to prevent accidental moves on precise clicks).
- **D-08:** **Group move is enabled** ‚Äî dragging any one selected markup moves all currently-selected markups by the same delta. Requires a batch `MoveMarkupsCommand` for undo.
- **D-09:** Movement threshold is **4px** ‚Äî reuse the existing rubber-band constant for consistency.

### Phase C ‚Äî Post-Commit Step-Level Undo (Phase 10 extension)

*Phase 10 gave step-level undo during in-progress drawing. This phase extends it to committed markups.*

- **D-10:** Ctrl+Z on a **committed** multi-point markup (linear, area, perimeter, wall) **re-opens it in drawing mode with all points intact** ‚Äî undoing just the commit, not removing the shape. The estimator can then add points, pop points with further Ctrl+Z, or re-commit with Enter. First Ctrl+Z after commit = "re-open"; subsequent Ctrl+Z = pop last point (same Phase 10 behavior).
- **D-11:** When the markup re-opens, show a **brief `ConfirmationToast`**: *"Shape re-opened ‚Äî continue drawing or press Enter to commit."* Uses the existing toast component (parent-owns-lifecycle pattern). Ensures first-time users understand the state change without blocking experienced users.
- **D-12:** Post-commit undo applies to all five multi-point tools: linear, area, perimeter, wall, and any future multi-point tool.

### Phase D ‚Äî Installer & Auto-Update (GAP-T2-06)

- **D-13:** Distribution and update channel: **GitHub Releases** ‚Äî free, zero infra, `electron-updater` polls the GitHub release API. New version = create a Release + upload installer + `latest.yml`.
- **D-14:** Auto-update notification: **prompt on launch if an update is available** ‚Äî dialog: "A new version is available. Install now?" Yes restarts and updates; No skips until next launch.
- **D-15:** Installer format: **unsigned NSIS `.exe`**, cost-free. Windows Defender SmartScreen will warn first-time users ("unknown publisher"); user clicks "More info ‚Üí Run anyway." Acceptable for internal/beta distribution. Code signing deferred until before client distribution (EV cert ~$300‚Äì400/yr; not in scope for v1.1).

### Phase E ‚Äî Pricing Column + Item Library (GAP-T1-03 + GAP-T1-05)

*Bundled into one phase ‚Äî library stores rates, pricing needs the library; both live under the Estimating ribbon tab.*

- **D-16:** Unit rate entry lives in a **dedicated panel within the Estimating ribbon tab** (currently has Export functionality; will gain a Pricing / Rates section). Separates "measuring" from "pricing" workflow. Estimating tab is the natural home for both rates and library management.
- **D-17:** Rate granularity: **per item name** (Claude's discretion). `rates: Record<string, number>` keyed by item name. Warn in the Estimating tab if the same item name appears with multiple UoM across pages.
- **D-18:** BOQ export adds **Rate ($/unit) and Cost ($) columns** after Quantity ‚Äî standard priced BOQ layout: `Item | Quantity | UoM | Rate | Cost`. Grand total Cost row at the bottom of each category and an overall grand total.
- **D-19:** **No currency symbol** in export or totals panel ‚Äî values are plain numbers. User applies currency formatting in Excel. Avoids currency assumptions; simple to implement.
- **D-20:** Item library is **per-project, embedded in the `.clmc` file** (additive optional field, no `formatVersion` bump needed). Library travels with the project file.
- **D-21:** Library management UI lives **under the Estimating ribbon tab** alongside rate entry ‚Äî same screen or tab panel, not a separate settings screen.
- **D-22:** Library stores **`{name, category, color, rate}`** ‚Äî when an estimator selects a library item during markup placement, name, category, color, and rate all auto-fill. Tightly couples library to pricing; both built in this phase.

### Phase F ‚Äî Markup Notes / Comments (GAP-T2-03)

- **D-23:** Notes support **both granularities**: a group-level note per item name (all placements share it) with a per-placement override for individual markups. `BaseMarkup` gains `notes?: string` (additive, no `formatVersion` bump). Item-name group notes live alongside the rates store.
- **D-24:** **Full visibility** ‚Äî three surfaces: (1) note icon badge on the markup canvas when a note exists; (2) hover tooltip shows the note text; (3) XLSX export gains a Notes column.

### Phase G ‚Äî Snap to Existing Geometry (GAP-T2-05)

- **D-25:** Snap targets: **existing markup vertices only** ‚Äî snap cursor to exact corner points of placed polylines and polygons. No midpoints, no grid in v1.1.
- **D-26:** Snap is **off by default**; a toolbar button (or ribbon toggle) enables it per session. Persisted in `localStorage clmc.ui`. Opt-in avoids surprising first-time users.

### Phase H ‚Äî Cross-Page Item Breakdown (GAP-T2-04)

- **D-27:** Panel UI: **separate "By Page" tab in the `TotalsPanelHeader`**. The existing summary view is the default tab; "By Page" tab shows all items with per-page quantity sub-rows.
- **D-28:** XLSX export: **separate "By Page" worksheet** in the same workbook. Main sheet unchanged. Second sheet: `Item | Page | Quantity | UoM` rows. Keeps the main sheet clean.

### Phase I ‚Äî PDF with Markups Export (GAP-T1-04)

- **D-29:** Include in v1.1 as **the final phase** ‚Äî after all other items are complete. Complexity: ~5 plans. pdf-lib draws Konva shapes (normalized page-space coords) onto the original PDF pages. Multi-page support required.

### Claude's Discretion

- **Q-11 (rate granularity):** Per item name chosen as the simplest correct model. Warn on UoM mismatch.
- **Q-27 (first v1.1 phase):** Scale Ratio Input first ‚Äî highest ROI per effort, smallest scope, standalone. Then Geometry Editing, then Post-Commit Undo, then Installer, then Pricing+Library, then Notes, Snap, Cross-page, PDF Export.
- **Q-28 (vertex + translate bundling):** One phase ‚Äî shared Konva drag infrastructure, single research pass, cleaner closure.
- **Q-29 (pricing + library bundling):** One phase ‚Äî library stores rates (D-22), both under Estimating tab (D-16/D-21), tightly coupled.

<specifics>
## Specific Ideas

### Scale Ratio Input ‚Äî Split input box UI
User explicitly requested: two separate numeric input fields with a colon separator ‚Äî `[ 1 ] : [ 100 ]`. The left field is locked to `1` (or editable but defaults to `1`). The right field is where the denominator goes. This eliminates all parsing ambiguity and matches how estimators read scale ratios from title blocks.

### Vertex Edit Exit ‚Äî Compound gesture
"Combination of Enter and clicking outside when not editing" ‚Äî means: Enter = commit (keyboard path); clicking outside the markup while no vertex handle drag is active = commit (mouse path); Escape = restore original and exit. The guard "when not editing" means the click-off path is suppressed during an active handle drag, preventing accidental commit mid-drag.

### Item Library Location ‚Äî Estimating ribbon tab
User confirmed: library management belongs under the Estimating tab ("does this fall under estimates? if yes let's do it under that"). Both rate entry and library management live in the Estimating ribbon tab panel, co-located as a single workflow: set rates ‚Üí save to library ‚Üí reuse across projects.

### Installer ‚Äî Cost-free is the constraint
User said "let's opt for a cost-free initiative." This locks the v1.1 installer to unsigned NSIS `.exe` (electron-builder default, no cert cost). SmartScreen warning is accepted for internal/beta use. EV code signing is explicitly out of scope for v1.1.

### Post-Commit Undo ‚Äî User intent is clear
User's exact words before the questionnaire: *"when a mark is placed when undo it erases the whole markup, can it be just a step-level undo?"* This confirms Q-30 option (b): Ctrl+Z on a committed shape re-opens it in drawing mode (all points intact). Q-31 = toast notification to signal the state change.

</specifics>
