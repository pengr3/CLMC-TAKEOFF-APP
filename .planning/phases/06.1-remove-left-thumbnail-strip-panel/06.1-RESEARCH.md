# Phase 06.1: Remove Left Thumbnail Strip Panel ‚Äî Research

**Researched:** 2026-05-12
**Domain:** React component removal / layout refactor (Electron renderer)
**Confidence:** HIGH ‚Äî all findings verified by direct codebase inspection

## Summary

Phase 6 shipped a three-column flex shell: `ThumbnailStrip (left) | CanvasViewport (center) | TotalsPanel (right)`. The decision to remove the thumbnail strip is a surgical deletion ‚Äî the component, its dedicated hook (`useThumbnailRender`), related state in `useUiPanels`, its `Splitter`, its local state in `App.tsx`, and its three test files must all go.

The left `Splitter` (between ThumbnailStrip and center column) is removed with the strip. The right `Splitter` (between center column and TotalsPanel) is **retained** ‚Äî the TotalsPanel still needs it. The `containerRef`/`containerWidth` mechanism in `App.tsx` is also retained because the right Splitter still uses it for its 50% max-width clamp.

`usePageLabels` is **retained** ‚Äî `CanvasHeaderBar` (center column) is its only other consumer and it's unaffected by this change.

`useUiPanels` is **partially edited** ‚Äî the `thumbnails` state shape, setters, storage key validation, and interface fields are removed. The `totals` and `collapsedCategories` parts remain untouched. The `use-ui-panels.test.ts` file must be updated to remove the thumbnail-specific test cases (four tests referencing `api.thumbnails` or `setThumbnailsOpen`).

**Primary recommendation:** Delete three files outright (`ThumbnailStrip.tsx`, `Thumbnail.tsx`, `useThumbnailRender.ts`), delete three test files outright, then make targeted edits to `App.tsx` and `useUiPanels.ts`, and update `use-ui-panels.test.ts`.

### localStorage Schema Change

Current `clmc.ui` key schema:
```json
{
  "thumbnails": { "open": true, "width": 140 },
  "totals": { "open": true, "width": 320 },
  "collapsedCategories": []
}
```

Target schema (thumbnails sub-key removed):
```json
{
  "totals": { "open": true, "width": 320 },
  "collapsedCategories": []
}
```

**Shape-check migration:** The defensive `readStorage()` function in `useUiPanels.ts` currently validates `parsed?.thumbnails?.open` and `parsed?.thumbnails?.width`. Both lines must be removed. Existing stored data in a user's browser that has the old 3-key shape will fail the shape check on first load (because the new check no longer expects the `thumbnails` key) ‚Äî this means it silently resets to DEFAULTS. That is the correct behavior per the existing T-06-01-01 hardening pattern. No explicit migration needed.

## Common Pitfalls

### Pitfall 1: Forgetting the left Splitter
**What goes wrong:** Only the `<ThumbnailStrip>` JSX block is removed but the `<Splitter side="left">` block immediately after it (lines 269‚Äì277) is left in place. The right Splitter still renders but the left one becomes a dangling divider with no panel to resize.
**How to avoid:** Remove both the ThumbnailStrip block AND the Splitter side="left" block together in one edit.

### Pitfall 2: Removing containerRef/containerWidth
**What goes wrong:** `containerRef` and `containerWidth` are removed as "thumbnail-related cleanup" but they are also passed to the right Splitter (`containerWidth={containerWidth}` at line 373). Removing them breaks the TotalsPanel resize 50% cap.
**How to avoid:** Keep `containerRef`, the `containerWidth` state, and the resize `useEffect`. Only remove the left Splitter's `containerWidth` usage.

### Pitfall 3: Removing the entire useUiPanels destructure
**What goes wrong:** The entire `useUiPanels()` destructure line is deleted, removing `totals, setTotalsOpen, setTotalsWidth` along with the thumbnail parts. TotalsPanel breaks.
**How to avoid:** Edit the destructure to keep `{ totals, setTotalsOpen, setTotalsWidth }` and remove only `thumbnails, setThumbnailsOpen, setThumbnailsWidth`.

### Pitfall 4: Leaving stale localStorage data unchecked
**What goes wrong:** The `readStorage()` shape check still validates `parsed?.thumbnails?.open` and `parsed?.thumbnails?.width`. Because the stored `clmc.ui` key still carries the `thumbnails` sub-key from Phase 6, the old shape will pass validation and `useUiPanels` will silently hold a stale `thumbnails` property even though nothing reads it. This isn't a runtime crash but leaves dead data in the hook's state.
**How to avoid:** Remove the thumbnail validation lines from `readStorage()` along with the `thumbnails` field from `UiState`. The shape check will then reject any stored key that still has the old shape (since `thumbnails` is no longer expected by the validator) and silently reset ‚Äî which is fine.

### Pitfall 5: Leaving thumbnail test stubs that import deleted files
**What goes wrong:** One or more of the three thumbnail test files is not deleted. The vitest run fails with import resolution errors for `ThumbnailStrip`, `Thumbnail`, or `useThumbnailRender`.
**How to avoid:** Delete all three test files before running `npm test`.

### Pitfall 6: Partial use-ui-panels.test.ts edit breaks remaining tests
**What goes wrong:** When trimming `use-ui-panels.test.ts`, the `returns DEFAULTS when localStorage has no entry` test is left partially intact with `api.thumbnails` assertions. TypeScript will not catch this ‚Äî Vitest will fail at runtime because `api.thumbnails` no longer exists on the hook return value.
**How to avoid:** Replace that test with a version that only asserts `totals` and `collapsedCategories` defaults. See Code Examples section.

## Code Examples

### Revised useUiPanels state shape
```typescript
// [VERIFIED: src/renderer/src/hooks/useUiPanels.ts]
export interface UiState {
  totals: { open: boolean; width: number }
  collapsedCategories: string[]
}

const DEFAULTS: UiState = {
  totals: { open: true, width: 320 },
  collapsedCategories: []
}

function readStorage(): UiState {
  try {
    const raw = window.localStorage.getItem(STORAGE_KEY)
    if (!raw) return DEFAULTS
    const parsed = JSON.parse(raw)
    if (
      typeof parsed?.totals?.open === 'boolean' &&
      typeof parsed?.totals?.width === 'number' &&
      Array.isArray(parsed?.collapsedCategories)
    ) {
      return parsed as UiState
    }
    return DEFAULTS
  } catch {
    return DEFAULTS
  }
}
```

## Runtime State Inventory

| Category | Items Found | Action Required |
|----------|-------------|-----------------|
| Stored data | `clmc.ui` localStorage key carries `thumbnails` sub-key on any machine that ran Phase 6 | No migration needed ‚Äî `readStorage()` silent-reset pattern handles stale shapes. When the updated shape-check runs, the stored key (which now includes the unvalidated `thumbnails` field) will either pass or fail the new narrower check. Because the new check does NOT validate `thumbnails`, a stored key like `{ "totals": {...}, "collapsedCategories": [], "thumbnails": {...} }` will actually pass validation (extra keys are ignored by the shape check) ‚Äî `totals` and `collapsedCategories` will be correctly restored. No data loss. |
| Live service config | None | ‚Äî |
| OS-registered state | None | ‚Äî |
| Secrets/env vars | None | ‚Äî |
| Build artifacts | None | ‚Äî |

**Key clarification on stored data:** The current `readStorage()` validates 5 typed fields. The new version validates 3. A stored payload with all 3 (plus the old `thumbnails` extras) passes ‚Äî extra keys are ignored by the `as UiState` cast. A stored payload missing any of the 3 core fields resets to DEFAULTS. This is the correct behavior.
