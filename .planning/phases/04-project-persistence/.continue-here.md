<decisions_made>

- **Phase 4.1 format: ZIP archive via jszip**
  - Current .clmc is plain JSON with external PDF reference (absolutePath + relativePath)
  - Problem: breaks when transferred to another PC — path doesn't exist, user gets MissingPdfModal and must manually Browse
  - User requirement: single file, any user on any PC can open it, no separate PDF needed
  - Four options evaluated: ZIP/jszip, custom binary, SQLite/sql.js, base64-in-JSON
  - **Chosen: ZIP/jszip** — pure JS (no native addon, Electron packaging stays simple), standard inspectable format, no size overhead (PDF bytes go in raw, already compressed), migration path clean

- **Migration strategy:** detect ZIP vs plain-JSON on open (magic bytes check), upgrade in-place — old .clmc files still open, user is prompted to re-save to upgrade format

- **Path resolution removed:** once PDF is embedded, absolutePath/relativePath/sha256 path-resolution logic is no longer needed. MissingPdfModal and HashMismatchModal become unnecessary (PDF is always present in the archive).

- **jszip over adm-zip:** jszip is async, better maintained, 5M+ weekly downloads, no native addon
</decisions_made>

<context>
Phase 04 ended successfully. The user noticed a real-world problem: if you save a .clmc and transfer it to a colleague's PC, the PDF reference breaks because the path is machine-specific. The relative path fallback only saves you if the PDF travels in the same folder. The user wants "one file, pass to anyone, works immediately." 

The ZIP/jszip approach solves this completely at the cost of file size (which is unavoidable — you need the PDF bytes somewhere). The user was enthusiastic about the approach ("why not"). Phase 4.1 is the natural next step before Phase 5 (BOQ Export), since export depends on project load working reliably for any user.
