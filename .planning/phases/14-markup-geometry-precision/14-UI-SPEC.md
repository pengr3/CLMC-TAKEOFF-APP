# Phase 14 — UI Design Contract

> Visual and interaction contract for Phase 14 (Markup Geometry Precision). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note (read first):** This phase is NOT a new HTML/React screen. It is CAD-style **on-canvas vector graphics** rendered with Konva on top of the existing PDF + markup canvas. The primary UI surface is transient overlay glyphs/handles drawn on the stage, plus two tiny HTML chrome surfaces that reuse the existing dark VS Code-style theme (a snap on/off status indicator and a blocked-commit message). Therefore the standard "Spacing / Typography / Color (60/30/10)" tables below describe the **already-established app chrome** that this phase merely reuses — the load-bearing new contract is in §§ "On-Canvas Glyph & Handle Contract", "Interaction & Cursor Contract", and "Feedback Surfaces". gsd-ui-checker should weight those three sections.

## Color

The existing app chrome palette (`constants.ts COLORS`) is reused verbatim — this phase does not alter the 60/30/10 split.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#1a1a1a` (`COLORS.dominant`) | Canvas backdrop, app background |
| Secondary (30%) | `#252526` (`COLORS.secondary`) | Status bar, panels, the snap-indicator pill, blocked-commit message surface |
| Accent (10%) | `#0078d4` (`COLORS.accent`) | See reserved list below |
| Destructive / problem | `#dc2626` (= `MARKUP_PALETTE[0]` red-600) | Self-intersection crossing highlight + blocked-commit message lead. **[INFERRED + rationale]** Reuses the existing palette's red-600 so the "problem" color already passes the WCAG-AA contrast guarantee baked into `MARKUP_PALETTE`; avoids inventing a parallel error red. |
| Warning (status) | `#e8a838` (`COLORS.warning`) | "Snap OFF" status text (reuses the same amber the status bar already uses for "Scale: Not Set") **[INFERRED + rationale]**: amber already means "attention, non-blocking" in this app's status bar. |

**Accent (`#0078d4`) reserved for, this phase:**
- The **snap indicator glyphs** (□ vertex, △ on-segment) — single accent stroke color so they read against both the PDF and any `MARKUP_PALETTE` markup stroke at all zooms. **[INFERRED + rationale]:** `#0078d4` is the established selection-ring accent (used by `HoverRing` selection variant via `COLORS.accent`); it is absent from `MARKUP_PALETTE` (closest is blue-600 `#2563eb`, visually distinct), so glyphs never camouflage against a same-colored markup. Accent blue also sits clearly against the dark `#1a1a1a` backdrop and typical white/grey PDF paper.
- The **snap-indicator pill** active border/glow when snapping is ON (border `#0078d4`).

**Destructive/problem (`#dc2626`) reserved for, this phase:**
- The highlighted offending crossing segment(s) on a blocked self-intersection commit (D-09).
- The lead word of the blocked-commit message.

**Snap-glyph contrast rule:** glyphs are drawn with the accent **fill at the core + a thin contrasting halo stroke** so they remain legible over both light and dark backgrounds. Halo color is chosen per-frame via `getContrastingInk()` against the markup color the glyph overlaps — reusing the existing `color-utils.ts` WCAG picker (threshold 0.179). **[INFERRED + rationale]:** the project already solved "mark must read on any background" for count-pin numbers with `getContrastingInk`; reusing it keeps one contrast system, not two.

## On-Canvas Glyph & Handle Contract  *(PRIMARY contract for this phase)*

All sizes are **screen pixels**, applied to Konva nodes as `value / currentZoom` so on-screen size is constant at every zoom (the load-bearing zoom-compensated overlay pattern; see `HoverRing.tsx`, `VertexHandleOverlay.tsx`). All glyphs/handles live on a **transient/overlay layer** (Layer 2 transient or the VertexHandleOverlay layer), above committed markups.

### Snap indicator glyphs (D-04)

Exactly two glyph types SHIP this phase. The ✕ glyph is RESERVED (drawn nowhere) so the legend stays stable when intersection snap lands later (D-06).

| Glyph | Snap type | Konva shape | Screen size | Stroke | Fill | Color | Opacity | listening |
|-------|-----------|-------------|-------------|--------|------|-------|---------|-----------|
| □ square | vertex / endpoint / close-the-loop start vertex | `Rect` (axis-aligned) | 12×12 px (`12/zoom`) | 2px (`2/zoom`) | none (transparent core) | accent `#0078d4` stroke + contrasting halo | 1.0 | **false** |
| △ triangle | nearest-point-on-segment | `RegularPolygon sides=3` (point up) | 14px circum-diameter (`14/zoom`) | 2px (`2/zoom`) | none | accent `#0078d4` stroke + contrasting halo | 1.0 | **false** |
| ✕ cross | intersection — **RESERVED, NOT SHIPPED** | — | — | — | — | — | — | — |

Glyph rules:
- **Exactly one** snap glyph is visible at a time — it marks the single active snap target under the cursor. No glyph renders when the cursor is not within tolerance of any candidate.
- The glyph is **centered on the snapped page-space point** (the coordinate the click will actually use), not the raw cursor — so the estimator sees exactly where the point will land.
- **Halo:** draw each glyph twice — a wider "halo" copy first (stroke `3.5/zoom`, color = `getContrastingInk()` of the underlying markup/PDF region) then the accent copy on top (stroke `2/zoom`). This mirrors the crosshair-cursor "black outline + white foreground" two-pass technique already in `CanvasViewport.tsx` (`CROSSHAIR_CURSOR`).
- **z-order:** snap glyph renders ABOVE the in-progress preview line and ABOVE the bulge handle, but it is non-listening so it never steals pointer events (same discipline as `HoverRing`/`PulseHighlight` `listening={false}`).
- Square vs triangle distinguishability from the existing **8px white vertex handle** (`VertexHandleOverlay`): the snap glyph is accent-colored + haloed + a different shape/size (12px square outline / 14px triangle vs 8px white-filled square). At a shared point both may briefly co-render; the accent stroke + halo reads on top of the white fill.

**Manual-ready caption — vertex snap (□):** *"A blue square means your cursor has locked onto an existing point — a corner or the end of a line. Click to place exactly on it."*
**Manual-ready caption — segment snap (△):** *"A blue triangle means your cursor has locked onto the nearest point along an existing line (not a corner). Click to drop a point right on that line."*
**Manual-ready caption — close-the-loop (□ on start vertex):** *"When tracing an area or perimeter, a blue square on your very first point means you're about to close the shape. Click to finish the loop."*

### Bulge handle — arc reshape (D-08)

The on-arc bulge handle is the edit gesture for arc curvature. It must be visually distinct from the 8px white square vertex handle.

| Property | Value | Source / rationale |
|----------|-------|--------------------|
| Konva shape | `Circle` (round — distinct from the square vertex handle) | **[INFERRED + rationale]:** circle vs square is the clearest "different kind of handle" signal at a glance; round also reads as "drag me along the curve." |
| Position | on the arc's current on-arc midpoint (the solved p2 the curve passes through) | spike 003 per-segment arc metadata carries the on-arc midpoint |
| Resting size | 9px diameter (`9/zoom`), i.e. radius `4.5/zoom` | **[INFERRED + rationale]:** ~1px larger than the 8px vertex square so the two handle families are visibly different but balanced; both stay sub-finger-tip at fit zoom. |
| Resting fill | accent `#0078d4` | Distinguishes from white vertex handles; reuses the established accent. |
| Resting stroke | 1.5px (`1.5/zoom`) white | matches the `VertexHandleOverlay` 1.5px stroke convention for handle borders |
| Hover state | grow to 12px diameter (`12/zoom`), stroke→`#ffffff` 2px | **[INFERRED + rationale]:** mirrors the "handle grows on hover" affordance; hit target is already larger than visual. |
| Active (dragging) state | filled `#0078d4`, stroke `#ffffff` 2px, plus a live dashed "chord+sagitta" guide drawn from the two endpoints through the dragged handle | **[INFERRED + rationale]:** echoes the existing dashed in-progress preview segment in `CanvasViewport.tsx`; shows the estimator how the curve re-solves as they drag. |
| Hit target | 18px square (`18/zoom`) hitWidth/hitHeight, listening **true** | **[INFERRED + rationale]:** mirrors `VertexHandleOverlay`'s 16px hit area for an 8px visual (hit > visual for precision, RESEARCH Finding 3); 18px keeps parity for the slightly larger handle. |
| Sagitta cap feedback | when the drag reaches the sagitta cap (max curvature before self-intersection risk), the dashed guide turns `#e8a838` (warning amber) and the handle stops following past the cap | **[INFERRED + rationale]:** D-09 says the sagitta cap should prevent most self-intersections; surfacing the cap as a soft amber stop teaches the limit without a hard error. Reuses the existing warning amber. |

**Manual-ready caption — bulge handle:** *"Each curved edge has a round blue handle at its midpoint. Drag it away from the straight line to deepen the curve, or toward it to flatten the curve. Drag past the safe limit and the guide turns amber — that's the maximum bend before the shape would fold on itself."*

**Manual-ready caption — endpoint re-solve (drag a vertex of an arc edge):** *"Dragging the square corner handle at either end of a curved edge re-bends the arc to pass through the new corner position — the curve follows the corner."*

## Interaction & Cursor Contract  *(PRIMARY contract for this phase)*

### Arc drawing — 3-click gesture (D-01)

| State | What the user does | On-canvas feedback |
|-------|--------------------|--------------------|
| Click 1 (start) | clicks the arc start point | start vertex placed; standard in-progress drawing continues |
| Move (after click 1) | moves cursor | **straight** dashed preview from start to cursor (existing behavior) **until** arc mode is active (see D-02) — then a live 2-point provisional preview |
| Click 2 (on-arc) | clicks a point the curve must pass through | provisional on-arc point captured; preview now shows a **live solved arc** through start → on-arc → cursor |
| Move (after click 2) | moves cursor toward the end | live arc preview re-solves every mousemove through the three points (start, on-arc, current cursor as end) — reuses spike-003 `solveCircle` |
| Click 3 (end) | clicks the arc end point | arc edge committed into the markup; if chain/arc-mode sticky, next edge continues |

- The **in-progress arc preview** is a Konva `Arc`/`Line`-sampled curve, drawn dashed in the markup's pending color on the transient layer, `listening=false`. **[INFERRED + rationale]:** matches the existing dashed straight-segment preview convention (`CanvasViewport.tsx` dashed preview lines) so arc-mode preview feels like the same family.
- Snapping (□/△ glyphs) applies to clicks 1 and 3 (endpoints) per D-05/D-07; the on-arc click (click 2) is a free point (it defines curvature, not a connection) — **[INFERRED + rationale]:** snapping an on-arc shaping point to other geometry would fight the gesture's purpose; spike 003c treats the middle click as a pure shaping input.

### Straight ↔ arc edge mode (D-02)

| Mechanism | Behavior | On-canvas / cursor affordance |
|-----------|----------|-------------------------------|
| Default | every new edge is **straight** | normal crosshair cursor; straight dashed preview |
| Momentary hold-key (one-off arc) | hold the arc key while drawing **one** edge → that edge is an arc, then auto-reverts to straight | while held: cursor gains a small arc badge **[INFERRED]**; preview switches to arc mode |
| Sticky toggle (run of arcs) | tap the toggle → "Arc next" stays ON for successive edges until toggled off | a persistent "Arc mode: ON" chip in the status indicator pill (see Feedback Surfaces) |
| Keybinding | hold-key `A` (proposed), sticky toggle TBD | **[OPEN]** — exact keys confirmed at plan time against Phase 9 ribbon/shortcut bindings to avoid collisions (D-02). |

- **Active arc-mode cursor:** when arc mode is active (held or sticky), the crosshair cursor is augmented with a tiny arc tick so the estimator always knows the next edge will curve. **[INFERRED + rationale]:** the app already swaps cursors by interaction state (`getCursor()` in `CanvasViewport.tsx` returns `CROSSHAIR_CURSOR` during drawing); an arc-mode variant fits that switch cleanly. If a per-state cursor proves heavy, fall back to relying solely on the status pill chip — recorded as an acceptable degradation.

**Manual-ready caption — arc mode:** *"Edges are straight by default. Hold `A` while drawing one edge to make just that edge a curve, or tap the Arc toggle to keep drawing curved edges until you turn it off. The cursor and the status bar both show when Arc mode is on."*

### Snapping activation & override (D-03)

| State | Trigger | Indicator |
|-------|---------|-----------|
| Snap ON (default) | app default | status pill shows "Snap: ON" with accent `#0078d4` border; snap glyphs appear when near a target |
| Snap suspended (1 point) | hold the suspend key while placing one point | status pill flips to a dimmed "Snap: held off"; **no snap glyph renders** while held; releasing restores ON |
| Snap OFF (persistent) | tap the snap toggle | status pill shows "Snap: OFF" in warning amber `#e8a838`; no glyphs render until toggled back |
| Keybindings | suspend hold-key `Alt` (proposed), persistent toggle `F3` (proposed) | **[OPEN]** — confirmed at plan time against Phase 9 bindings (D-03, flagged as the one open detail). |

- **Snap-suspended vs snap-off distinction:** suspended = momentary/dimmed (returns to ON on key release); off = sticky/amber (stays until re-toggled). The absence of any snap glyph is the primary on-canvas signal in both cases; the status pill disambiguates which one is active. **[INFERRED + rationale]:** glyph-presence is the in-flow signal the estimator watches; the pill is the persistent ground truth.

**Manual-ready caption — snapping:** *"Snapping is on by default — your cursor jumps to nearby points and lines and shows a blue marker. Hold `Alt` to ignore snapping for a single click, or press `F3` to switch snapping off entirely (the status bar shows OFF in amber). Press `F3` again to turn it back on."*

### Snap-target rule (D-07) — interaction invariant

- Snap freely to **other** markups' vertices and segments.
- For the **in-progress or actively-edited** markup, snap **only** to its own start vertex (close-the-loop, □ glyph) — never to its own intermediate vertices, never to the vertex currently being dragged.
- Applies in **both** placement and Phase-12 vertex/body-drag editing.

### Self-intersection blocked commit (D-09)

| Step | Behavior |
|------|----------|
| User attempts to commit an area/perimeter whose boundary self-intersects | **Commit is BLOCKED.** The markup stays in edit/drawing mode. |
| Visual | the offending crossing segment(s) are highlighted in problem red `#dc2626` (drawn on the transient layer, slightly thicker than the markup stroke: `(markupStroke + 2)/zoom`); the crossing point may show a small ✕-shaped red mark **[INFERRED — distinct from the reserved blue intersection-snap ✕ by color]** |
| Message | the blocked-commit message surface appears (see Feedback Surfaces) explaining the problem + the fix |
| Recovery | the estimator drags vertices / the bulge handle to remove the crossing, then re-commits |

**Manual-ready caption — blocked commit:** *"If an area or perimeter outline crosses over itself, the app won't let you finish it — a wrong shape would report a wrong quantity. The crossing is highlighted in red. Drag the corners or curve handles apart until the red clears, then finish again."*

## Feedback Surfaces  *(HTML chrome — reuses existing theme)*

Two small non-canvas surfaces. Both reuse the dark VS Code-style chrome (`COLORS.secondary #252526` background, `COLORS.border #3c3c3c`, 13px/400 text) read from `StatusBar.tsx` and the existing `ConfirmationToast` pattern.

### 1. Snap / arc-mode status indicator (pill)

- **Placement:** in the existing bottom **StatusBar**, appended as a new segment after the Scale segment, separated by the existing 1px `#3c3c3c` `Divider`. **[INFERRED + rationale]:** the status bar already aggregates persistent mode info (page, zoom, scale) at 13px/400; snap/arc state belongs in the same row rather than a new floating element.
- **Content:** `Snap: ON|OFF|held off` (+ accent dot) and, when arc mode active, an `Arc` chip.
- **States:** ON → text `#cccccc`, accent `#0078d4` status dot. Held-off → dimmed `#808080`. OFF → `#e8a838` (warning amber). Arc chip → accent `#0078d4` background, `#ffffff` text.
- `aria-live="polite"` (matches the Scale segment in `StatusBar.tsx`).

### 2. Blocked-commit message

- **Pattern:** reuse the existing **`ConfirmationToast`** presentational pattern (parent owns dismissal lifecycle — STATE.md "ConfirmationToast is pure presentational (no setTimeout)"). It is NOT auto-dismissed by a timer; it clears when the self-intersection is resolved and the user re-commits, or on cancel.
- **Surface:** `#252526` background, `1px #3c3c3c` border, 24px internal padding, 13px text. Lead word in red `#dc2626` weight 600.
- **Copy:** see Copywriting Contract.
- **Placement:** anchored near the offending markup centroid (screen-space), matching how the app already positions markup popups (e.g. `finishPolygon` centroid+20px). **[INFERRED + rationale]:** keeps the message next to the problem the estimator must fix.

## Reuse / Do-Not-Reinvent Checklist (for executor)

- Stroke widths, radii, offsets → divide by `currentZoom` (every glyph/handle). Source: `HoverRing.tsx`, `PulseHighlight.tsx`, `VertexHandleOverlay.tsx`.
- Transient overlays → `listening={false}` (snap glyphs, in-progress arc preview, crossing highlight). Interactive handle (bulge) → `listening={true}` with hit area larger than visual.
- Contrast → `getContrastingInk()` from `color-utils.ts` (threshold 0.179) for glyph halos. Do not invent a new contrast function.
- Accent → `COLORS.accent #0078d4`. Problem red → `MARKUP_PALETTE[0] #dc2626`. Warning amber → `COLORS.warning #e8a838`. Do not introduce new hexes.
- Cursor variants → extend the existing `getCursor()` switch + `CROSSHAIR_CURSOR` two-pass SVG technique in `CanvasViewport.tsx`.
- Status text → mirror `StatusBar.tsx` (13px/400, `#cccccc`, `#3c3c3c` dividers, 16px padding, `aria-live="polite"`).
- Message → mirror `ConfirmationToast` parent-owned lifecycle (no internal setTimeout).
- Layer placement → transient/overlay layer above committed markups; bulge handle on the VertexHandleOverlay-style layer so it intercepts events before markup bodies.
