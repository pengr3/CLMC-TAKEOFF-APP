# Phase 7 ŌĆö UI Design Contract

> Visual and interaction contract for **Canvas Workspace UX and Markup Editing Fixes** (canvas gutter fix, post-commit markup editing, totals panel cleanup, Set Scale dropdown overflow fix, category deduplication UX). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase is **net-additive bug-fix work** on the existing dark-theme Electron + React 19 + Konva chrome. **Zero new design tokens, zero new fonts, zero new color hex literals, zero new icon imports** are introduced. The five fixes all extend or modify components that already exist (`CanvasViewport`, `MarkupContextMenu`, `MarkupNamePopup`, `TotalsPanel`, `TotalsCategoryBlock`, `CalibrationDialog`, `CategoryAutocomplete`). This contract carries forward the Phase 6 UI-SPEC verbatim for all unchanged surfaces and prescribes only the deltas.

## Layout Contract

Phase 7 does **not** change the three-column shell from Phase 6 or the two-column shell from Phase 6.1. The only structural CSS change is **inside `CanvasViewport`** ŌĆö its root container fills its flex parent correctly.

### Canvas Root Fill (D-01 / D-02)

**Current (broken) layout:**
```
App.tsx
  ŌööŌöĆ <div style={{ flex: 1, position: 'relative' }}>   ŌåÉ center column; no explicit height
       ŌööŌöĆ CanvasViewport
            ŌööŌöĆ <div ref={containerRef} style={{ width: '100%', height: '100%' }}>   ŌåÉ percentage on undefined parent height = 0 ŌåÆ Stage falls back to {800, 600}
                 ŌööŌöĆ <Stage width={containerSize.width} height={containerSize.height} />
```
Result: Stage locks at 800├Ś600; PDF page positioned top-left; dark gutters fill the rest of the actual available space.

**Fixed layout:**
```
App.tsx
  ŌööŌöĆ <div style={{ flex: 1, position: 'relative' }}>   ŌåÉ unchanged
       ŌööŌöĆ CanvasViewport
            ŌööŌöĆ <div ref={containerRef} style={{ position: 'absolute', inset: 0 }}>   ŌåÉ fills parent's layout-computed bounds; no percentage-height dependency
                 ŌööŌöĆ <Stage width={containerSize.width} height={containerSize.height} />   ŌåÉ ResizeObserver now reads true available area
```
Result: Stage fills the entire center column; PDF page centered using the existing `(containerSize.width - pageWidth * fitScale) / 2` math (no math changes ŌĆö D-03).

**Rule:** Any positioned children inside the CanvasViewport root (`CalibrationDialog`, `MarkupNamePopup`, `MarkupContextMenu`, `MarkupTooltip`, `ScalePopup`, `ConfirmationToast`) already use `position: absolute` with their own anchoring. The container change to `inset: 0` does NOT regress any of their positions because the container retains its `position: relative` semantic (a `position: absolute` div with `inset: 0` is itself a positioning context for its `position: absolute` children).

### CalibrationDialog Dropdown Overflow Fix (D-11)

**Investigation result (researcher trace through CanvasViewport.tsx and CalibrationDialog.tsx):**
- `CalibrationDialog` mounts inside `CanvasViewport`'s root div via `position: absolute; inset: 0` with an internal flex centering wrapper.
- The native `<select>` element renders its dropdown list using OS chrome (Chromium-native dropdown popout). On Chromium 134 the dropdown popout **does** escape ancestor `overflow: hidden` because it's drawn outside the document flow as an OS popup.
- **Reported clipping is almost certainly a stacking-context issue, not an overflow issue:** the dropdown popout opens behind the modal backdrop `rgba(0,0,0,0.55)` because the dialog frame's z-index does not include `isolation: isolate` and the popout inherits a lower stacking context.

**Fix (priority order ŌĆö planner picks the first that works):**

1. **Option A ŌĆö Stacking-context hardening (preferred, zero-component-rewrite):**
   - Add `isolation: isolate` to the `CalibrationDialog` outer overlay div (the `position: absolute; inset: 0; zIndex: 10` div).
   - Add `position: relative; zIndex: 11` to the dialog frame div if needed to ensure the native dropdown popout draws above the backdrop.
   - Acceptance: the dropdown popout is fully visible, list items are clickable, no scrollbar appears inside the popout when there are Ōēż8 options.

2. **Option B ŌĆö Explicit z-index on the select (fallback):**
   - Add `position: relative; zIndex: 1` to the `<select>` itself. (Chromium sometimes requires the select to have a positioned context for its popout to inherit z-index correctly.)

3. **Option C ŌĆö Custom dropdown (last resort, only if A and B fail):**
   - Replace the native `<select>` with a button-triggered list styled to match the existing inline-style dark theme. Button styling matches the existing inputs (`COLORS.dominant` background, `COLORS.border` 1px, `COLORS.textPrimary` text). Open state shows a popover at the button's position with `COLORS.secondary` background, `COLORS.border` 1px, `boxShadow: '0 4px 12px rgba(0, 0, 0, 0.4)'`, max-height 200px, scroll, each row at `autocompleteRowHeight` (24px). Selected row gets a 2px `COLORS.accent` left border. Mirrors the `CategoryAutocomplete` interaction pattern.

**The CalibrationDialog modal itself does not change shape, size, or copy** ŌĆö except the secondary button label, which moves from generic `Cancel` to the specific `Discard Scale` (see Copywriting Contract). Only the dropdown rendering and the secondary button label are touched.

**Click `Save Changes`:**
- The popup confirms with payload `{ name, categoryName, color }`.

### Category Deduplication UX (D-12, D-13)

**Keyboard navigation (D-12) inside `CategoryAutocomplete`:**

- **`ArrowDown` / `ArrowUp`** on the category input (forwarded via `MarkupNamePopup` `onKeyDown`):
  - First `ArrowDown` when no row is highlighted ŌåÆ highlight index 0.
  - Subsequent presses advance/retreat through `filtered.length + (showCreateOption ? 1 : 0)` rows.
  - Wrap-around: at the last row, `ArrowDown` wraps to row 0; at row 0, `ArrowUp` wraps to the last row. (Locked: standard combobox idiom.)
  - The highlighted row gets `background: COLORS.hoverSurface` AND a 2px left border in `COLORS.accent`. Mouse-hovered rows get `background: COLORS.hoverSurface` only (no left border). This visual distinction lets a user keyboard-navigate without losing track when the mouse is also over a different row.
  - The highlighted row auto-scrolls into the visible list area: `rowElement.scrollIntoView({ block: 'nearest', behavior: 'auto' })` (no smooth scroll ŌĆö `auto` matches Electron desktop chrome expectations).

- **`Enter`** on the category input with a highlighted row:
  - Invokes the same `onSelect(categoryName)` callback that the row's `onMouseDown` invokes.
  - Selection commits the canonical name back to the category input field.
  - Closes the suggestion list (existing `showCategoryList` state transitions to `false`).
  - **Does NOT** submit the popup form. The user can still review name/color before clicking `Save Changes` / `Start Count` / `Save Markup`.

- **`Enter`** on the category input with NO highlighted row AND no suggestions visible:
  - Default behavior: submits the popup (same as a typing-in-Name `Enter`). Existing behavior unchanged.

- **`Escape`** on the category input:
  - Closes the suggestion list (existing behavior unchanged).
  - Does NOT cancel the popup.

- **`Tab`** on the category input:
  - Existing behavior (focus moves to the next form element) ŌĆö unchanged.

- **Highlighted-index reset:**
  - When `query` changes (user types), reset `highlightedIndex` to `-1` (no highlight). Prevents stale-index out-of-bounds against the new filtered list.
  - When the list closes (Escape, selection, outside-click), reset to `-1`.
  - When mouse hovers a row, **do not** sync `highlightedIndex` ŌĆö mouse-hover and keyboard-highlight are independent states so the user can see "you're moving the mouse but the keyboard is still here." (Locked discretion.)

**Canonical-name substitution (D-13) inside `MarkupNamePopup.handleConfirm`:**

- Before invoking `onConfirm`, run:
  ```typescript
  const typed = categoryName.trim()
  const canonical = typed === '' ? '' : (findCategoryByName(typed)?.name ?? typed)
  onConfirm({ name: trimmedName, categoryName: canonical, color: selectedColor })
  ```
- If `findCategoryByName` returns a stored category, its canonical `name` (with the original capitalisation) replaces the user-typed string in the payload.
- Applies to all three modes (`'count-pre'`, `'save-after'`, `'edit'`).
- Result: `getOrCreateCategory(canonical)` at the consumer site (CanvasViewport, useMarkupTool, or EditMarkupCommand) hits the existing category branch via case-insensitive match, returning the existing `categoryId`. No new category record is created. No typo-based duplicate.

**Empty string handling:** if the user leaves the category input blank, the payload's `categoryName` stays the empty string. The downstream `getOrCreateCategory` maps empty to `'Uncategorized'` (existing behavior, Phase 3 STATE decision).
