# Plan 12-01: Wave 1 — Foundation: Types + Store Actions + viewerStore Extension

## Goal

Extend `markup.ts` with two new `MarkupCommand` union members, add `moveVertex` and `moveMarkups` store actions with undo/redo branches to `markupStore`, and add `vertexEditMarkupId` state to `viewerStore` — making the Wave 0 RED tests GREEN.

## Context

This is the data-model and state foundation that every subsequent plan builds on. No UI or Konva code is touched here. After this plan:

- `MarkupCommand` accepts `move-vertex` and `move-markups`
- `markupStore` exposes `moveVertex(markupId, page, vertexIndex, newPoint)` and `moveMarkups(moves[])` with symmetric undo/redo
- `viewerStore` exposes `vertexEditMarkupId`, `setVertexEditMarkupId`, `clearVertexEdit`
- All three Wave 0 test files pass (GREEN)

The `move-markups` command normalises count pin moves as `oldPoints: [markup.point]` / `newPoints: [newPoint]`, so the undo reducer can handle all markup types uniformly.

## Tasks

### T1: Extend `MarkupCommand` union in `markup.ts`

**Type:** edit
**File:** `src/renderer/src/types/markup.ts`

**What:**

Append two new union members to the `MarkupCommand` type (after the `edit-markup` member, before the closing semicolon). Do NOT modify any existing member.

First new member — `move-vertex`:
```
type: 'move-vertex'
markupId: string
page: number
vertexIndex: number        // 0-based index into markup.points[]
oldPoint: StagePoint
newPoint: StagePoint
```

Second new member — `move-markups`:
```
type: 'move-markups'
moves: Array<{
  markupId: string
  page: number
  oldPoints: StagePoint[]  // [markup.point] for count pins, markup.points for others
  newPoints: StagePoint[]  // moved positions
}>
```

`StagePoint` is already imported at line 1 from `'../hooks/useCalibrationMode'` — no new import needed.

**Why:** `MarkupCommand` must be extended before the store actions can reference these types. Additive union extension per the STATE.md "additive schema fields" decision — no `formatVersion` bump needed (commands are in-memory only).

### T2: Add `moveVertex` and `moveMarkups` actions to `markupStore.ts`

**Type:** edit
**File:** `src/renderer/src/stores/markupStore.ts`

**What:**

**Step A — Add to the `MarkupStoreState` interface** (after the `editMarkup` signature):
```
moveVertex: (markupId: string, page: number, vertexIndex: number, newPoint: StagePoint) => void
moveMarkups: (moves: Array<{ markupId: string; page: number; oldPoints: StagePoint[]; newPoints: StagePoint[] }>) => void
```

Import `StagePoint` from `'../hooks/useCalibrationMode'` at the top of the file (add to the existing import line or add a new one).

**Step B — Implement `moveVertex` action** (after `editMarkup` implementation, before `getColorForName`):

Follow the `editMarkup` pattern: enter `set((s) => { ... })` directly (no category resolution needed). Inside:
- Look up `pageList = s.pageMarkups[page] ?? []`
- Find `target = pageList.find(m => m.id === markupId)`
- Defensive no-op: `if (!target) return s`
- For non-count markup: build `updatedPoints` array by replacing `target.points[vertexIndex]` with `newPoint` (spread: `[...target.points.slice(0, vertexIndex), newPoint, ...target.points.slice(vertexIndex + 1)]`)
- Build `updated: Markup = { ...target, points: updatedPoints }` (TypeScript will narrow correctly because only `count` markups lack `points`, and `move-vertex` must not be called on count markups — but add no explicit guard here; the caller is responsible)
- Build the command: `{ type: 'move-vertex', markupId, page, vertexIndex, oldPoint: target.points[vertexIndex], newPoint }`
- Return `{ pageMarkups: { ...s.pageMarkups, [page]: nextPageList }, undoStack: pushCommand(s.undoStack, cmd), redoStack: [] }`

**Step C — Implement `moveMarkups` action** (after `moveVertex`):

Follow the `deleteGroup` multi-page iteration pattern. Enter `set((s) => { ... })`. Inside:
- Guard: `if (moves.length === 0) return s`
- Build `nextPageMarkups = { ...s.pageMarkups }`
- Iterate `moves`. For each `move`, find the markup in `nextPageMarkups[move.page]`.
- If `markup.type === 'count'`: apply `newPoints[0]` as `markup.point`. Build updated markup as `{ ...markup, point: move.newPoints[0] }`.
- Otherwise: apply `newPoints` as `markup.points`. Build updated markup as `{ ...markup, points: move.newPoints }`.
- Replace the markup in the page list using `.map`.
- Build the command: `{ type: 'move-markups', moves: moves.map(m => ({ markupId: m.markupId, page: m.page, oldPoints: m.oldPoints, newPoints: m.newPoints })) }`
- Return updated state with `pushCommand(s.undoStack, cmd)` and `redoStack: []`.

**Step D — Add undo/redo branches for `move-vertex` and `move-markups`**:

In the `undo()` reducer, insert TWO new branches BEFORE the `delete-group` branch (which has the comment "MUST come BEFORE the `cmd.markup.page` access below"). Both new branches must also come before that fallthrough.

`move-vertex` undo branch:
- `if (cmd.type === 'move-vertex')`: find the markup on `cmd.page`, replace `points[cmd.vertexIndex]` with `cmd.oldPoint`. Return `{ pageMarkups: updated, undoStack: s.undoStack.slice(0, -1), redoStack: [...s.redoStack, cmd] }`.

`move-markups` undo branch:
- `if (cmd.type === 'move-markups')`: iterate `cmd.moves`, for each restore `oldPoints`. Count pins: `{ ...markup, point: move.oldPoints[0] }`. Others: `{ ...markup, points: move.oldPoints }`. Return `{ pageMarkups: updated, undoStack: s.undoStack.slice(0, -1), redoStack: [...s.redoStack, cmd] }`.

In the `redo()` reducer, insert matching branches in the same position (before `delete-group`):

`move-vertex` redo: same as the forward `moveVertex` action — replace `points[cmd.vertexIndex]` with `cmd.newPoint`. Return with `pushCommand(s.undoStack, cmd)` and `redoStack: s.redoStack.slice(0, -1)`.

`move-markups` redo: re-apply `newPoints` for each move. Return with `pushCommand`.

**Why:** These actions are the backbone of all Phase 12 geometry edits. The symmetric undo/redo pattern mirrors every existing command type, keeping the undo stack simple and testable.

**Type:** edit
**File:** `src/renderer/src/stores/viewerStore.ts`

**What:**

**Step C — Clear `vertexEditMarkupId` on all page-navigation actions.** In `setPage`, `nextPage`, and `prevPage`, add `vertexEditMarkupId: null` to each `set({...})` call alongside the existing `selectedMarkupIds: []`. In `setFile`, `resetViewer`, and `hydrate`, also set `vertexEditMarkupId: null`.

This mirrors the existing `selectedMarkupIds: []` lifecycle — vertex edit is page-scoped (Pitfall 3 in RESEARCH.md).

**Step D — Update the `beforeEach` reset in `src/tests/vertex-edit-mode.test.ts`** — this test file was written in Wave 0 with the new fields. The `setState` call there already includes `vertexEditMarkupId: null`, so no test file changes are needed. TypeScript will stop complaining about the missing field once Step A is done.

**Why:** `vertexEditMarkupId` drives React re-renders (the handle overlay mounts/unmounts based on it), so it must live in Zustand store state rather than a bare ref.
