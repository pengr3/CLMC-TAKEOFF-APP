# Plan 12-02: Wave 2 — Markup Component Drag Props + VertexHandleOverlay Component

## Goal

Extend all five markup component interfaces with optional drag props (`onMarkupMouseDown`, `overridePoints`/`overridePoint`) and build the `VertexHandleOverlay` component as a standalone Konva Group that renders zoom-compensated 8×8px square handles for one markup's vertices.

## Context

This plan makes the markup renderers "drag-aware" without breaking any existing callsite. All new props are optional — callers that do not pass them get identical behavior. The `VertexHandleOverlay` component is self-contained here and wired into `CanvasViewport` in Wave 3.

Two key architectural rules from RESEARCH.md:
- Handle Rects must have `listening={true}` (unlike HoverRing/PulseHighlight which are `listening={false}`)
- Handle size = `8 / currentZoom`, stroke = `1.5 / currentZoom` — zoom-compensated, constant on screen at all zoom levels (D-05)
- Each Rect gets `name={`handle-${i}`}` for named hit detection in `handleStageMouseDown`

## Tasks

### T1: Add drag props to all five markup component interfaces and render bodies

**Type:** edit
**Files:**
- `src/renderer/src/components/markup/LinearMarkup.tsx`
- `src/renderer/src/components/markup/AreaMarkup.tsx`
- `src/renderer/src/components/markup/PerimeterMarkup.tsx`
- `src/renderer/src/components/WallMarkup.tsx`
- `src/renderer/src/components/markup/CountPinMarkup.tsx`

**What:**

**For all five components — add to each props interface:**
```typescript
/** Called on mousedown on the markup body (Group), forwarding the markup id.
 *  CanvasViewport reads this ref to start body-drag before the Stage-level handler fires. */
onMarkupMouseDown?: (id: string) => void
```

**For LinearMarkup, AreaMarkup, PerimeterMarkup, WallMarkup — additionally add:**
```typescript
/** When set, render these points instead of markup.points (live drag preview). */
overridePoints?: StagePoint[]
```

**For CountPinMarkup — additionally add:**
```typescript
/** When set, render this point instead of markup.point (live drag preview). */
overridePoint?: StagePoint
```

**In each component's render function:**

For `LinearMarkup`, `AreaMarkup`, `PerimeterMarkup`, `WallMarkup`: resolve the effective points array at the top of the render function before any usage of `markup.points`:
```typescript
const effectivePoints = overridePoints ?? markup.points
```
Replace all references to `markup.points` in the Konva geometry (the `<Line points={...} />` and any centroid/midpoint calculations) with `effectivePoints`. Label chip position calculations that use `markup.points` should also use `effectivePoints`. Do NOT replace `markup.points` references in the `isHidden` composite key or the `markup.id` / `markup.color` / `markup.name` accesses — those are metadata reads, not geometry.

For `WallMarkup` specifically: both the primary `<Line>` and the offset hairline use `markup.points`. Replace both with `effectivePoints`. The `wallHeight` and offset calculation remain unchanged.

For `CountPinMarkup`: resolve the effective point:
```typescript
const effectivePoint = overridePoint ?? markup.point
```
Replace the `<Circle x={markup.point.x} y={markup.point.y} .../>` and the `<Text x={markup.point.x - PIN_RADIUS_WORLD} y={markup.point.y - NUMBER_FONT_WORLD / 2} .../>` to use `effectivePoint.x` and `effectivePoint.y`.

**In each component's `<Group>` element — add the `onMouseDown` handler:**
```typescript
onMouseDown={() => onMarkupMouseDown?.(markup.id)}
```
Place this alongside the existing `onClick`, `onMouseEnter`, `onMouseLeave`, and `onContextMenu` handlers on the Group. The prop is optional — existing callsites that don't pass it see no behavior change.

**Why:** Making all five components drag-aware in a single task ensures no markup type is missed (Pitfall 8 in RESEARCH.md). The `onMarkupMouseDown` prop enables the CanvasViewport bubbling-order trick in Wave 3.

### T2: Create `src/renderer/src/components/markup/VertexHandleOverlay.tsx`

**Type:** create
**File:** `src/renderer/src/components/markup/VertexHandleOverlay.tsx`

**What:**

Create a new React component exported as `VertexHandleOverlay`. It receives the following props:
```typescript
export interface VertexHandleOverlayProps {
  markup: Markup                       // the markup currently in vertex-edit mode
  currentZoom: number
  /** Called when a handle's mousedown fires, with the vertex index. */
  onHandleMouseDown?: (vertexIndex: number) => void
}
```

Import `Rect, Group` from `'react-konva'`. Import `Markup` from `'../../types/markup'`. Import `StagePoint` from `'../../hooks/useCalibrationMode'`.

**Implementation:**

Module-level constants (zoom-compensated at call-site, constants stay as screen pixels):
```typescript
const HANDLE_SIZE_PX = 8    // screen pixels — D-05
const HANDLE_STROKE_PX = 1.5
```

Inside the component:
```typescript
const handleSize = HANDLE_SIZE_PX / currentZoom
const handleStroke = HANDLE_STROKE_PX / currentZoom
const hitSize = 16 / currentZoom   // larger hit area for precision (RESEARCH Finding 3)
```

Determine the points to render handles for:
```typescript
const points: StagePoint[] = markup.type === 'count'
  ? []   // count pins have no vertex handles — translate-only (D-09 rule)
  : markup.points
```

Render a `<Group>` (no `listening` prop needed on Group — defaults to `true`). Inside, map over `points` and render one `<Rect>` per vertex:
```typescript
<Rect
  key={i}
  name={`handle-${i}`}        // named for hit detection — RESEARCH Finding 4
  x={p.x - handleSize / 2}
  y={p.y - handleSize / 2}
  width={handleSize}
  height={handleSize}
  fill="#ffffff"               // D-05: white fill
  stroke={markup.color}        // D-05: colored border matching markup color
  strokeWidth={handleStroke}
  hitWidth={hitSize}           // wider hit area than visual
  hitHeight={hitSize}
  listening={true}             // CRITICAL: handles MUST accept pointer events (Pitfall 6)
  onMouseDown={(e) => {
    e.cancelBubble = true      // prevent Stage-level handler from also firing
    onHandleMouseDown?.(i)
  }}
/>
```

**Important comments to include in the file:**
- Above the `listening={true}` line: `// CRITICAL: handles MUST be listening=true — unlike HoverRing/PulseHighlight (Pitfall 6 in 12-RESEARCH.md)`
- Above `e.cancelBubble = true`: `// Stop propagation so Stage onMouseDown does not also fire for this handle click`

The component returns the `<Group>` directly. If `points.length === 0` (count pin), return `null`.

**Why:** The `VertexHandleOverlay` is built standalone here so Wave 3 can wire it into `CanvasViewport` without simultaneously implementing it. The `name={`handle-${i}`}` prop is critical for the hit-detection pattern in Wave 3's `handleStageMouseDown`.
