# Plan 12-03: Wave 3a — CanvasViewport Foundation: Refs, Vertex-Edit Activation, Keyboard Handlers

## Goal

Add the new drag refs and drag-preview local state to `CanvasViewport`, wire the `vertexEditMarkupId` store subscriptions, implement vertex-edit activation via the second-click `handleMarkupClick` extension, and add Enter/Escape keyboard handlers for commit and cancel — without yet implementing body-drag or the vertex-drag move event flow (those come in Wave 3b).

## Context

`CanvasViewport.tsx` is the most complex file in the codebase. To keep each agent context window focused, Phase 12 splits Wave 3 into three sub-plans:
- **12-03 (this plan):** Refs, state, store subscriptions, `handleMarkupClick` extension, Enter/Escape handlers, `VertexHandleOverlay` layer mount
- **12-04:** Translate drag (body drag): `markupBodyDownRef`, `bodyDragRef`, `bodyDraggedRef`, `handleStageMouseDown` body-drag branch, `handleStageMouseMove` body-drag branch, `handleStageMouseUp` body-drag commit, `handleStageClick` body-drag suppression
- **12-05:** Vertex drag: `vertexDragRef`, `handleStageMouseDown` handle-drag branch, `handleStageMouseMove` vertex-drag branch, `handleStageMouseUp` vertex-drag commit, click-outside commit

Each plan can read only its own concerns. The plans are sequential (each `depends_on` the prior), not parallel.

### T2: Extend `handleMarkupClick` for second-click vertex-edit activation

**Type:** edit
**File:** `src/renderer/src/components/CanvasViewport.tsx`

**What:**

Replace the existing `handleMarkupClick` callback (lines ~424–430) with the extended version per RESEARCH Finding 5:

```typescript
const handleMarkupClick = useCallback(
  (id: string) => {
    if (activeTool !== 'select') return

    // D-04: second click on an already-selected single markup enters vertex edit mode.
    // Guard: count pins translate-only — no vertex edit (D-09 rule).
    const markup = pageMarkups.find((m) => m.id === id)
    if (!markup) return

    if (markup.type !== 'count' && selectedMarkupIds.length === 1 && selectedMarkupIds[0] === id) {
      // Store the original points for Escape-restore BEFORE entering vertex edit.
      vertexEditOriginalRef.current = [...markup.points]
      setVertexEditMarkupId(id)
      return
    }

    // First click (or different markup): clear vertex edit, select the clicked markup.
    clearVertexEdit()
    vertexEditOriginalRef.current = null
    setSelectedMarkupIds([id])
  },
  [activeTool, selectedMarkupIds, pageMarkups, setSelectedMarkupIds, setVertexEditMarkupId, clearVertexEdit]
)
```

**Why:** The second-click detection `selectedMarkupIds.length === 1 && selectedMarkupIds[0] === id` is the D-04 implementation. Count pins short-circuit to plain selection. The original-snapshot is captured at activation time so `cancelVertexEdit` can restore without any store round-trip.

### T3: Add Enter/Escape keyboard handlers for vertex edit commit/cancel

**Type:** edit
**File:** `src/renderer/src/components/CanvasViewport.tsx`

**What:**

**Step A — Define `commitVertexEdit` and `cancelVertexEdit` callbacks** (after the `handleMarkupClick` callback, before the toast state):

```typescript
// Phase 12: commit vertex edit — clears drag preview and exits vertex edit mode.
// vertex dispatch happens in handleStageMouseUp; this is cleanup only
const commitVertexEdit = useCallback(() => {
  setDragPreview(null)
  clearVertexEdit()
  vertexEditOriginalRef.current = null
}, [clearVertexEdit, setDragPreview])

// Phase 12: cancel vertex edit — restores original points via overridePoints prop.
const cancelVertexEdit = useCallback(() => {
  setDragPreview(null)
  clearVertexEdit()
  vertexEditOriginalRef.current = null
}, [clearVertexEdit, setDragPreview])
```

Note: `commitVertexEdit` is cleanup only — it clears the drag preview and exits vertex edit mode. The actual vertex-position dispatch ONLY happens in `handleStageMouseUp` (Plan 12-05). Do NOT add any store dispatch or forEach loop here.

```jsx
// Phase 12: vertex-edit handles layer — ABOVE Layer 1b, listening=true so handle Rects
// intercept pointer events before the markup body beneath them.
const vertexHandleLayer = (() => {
  if (vertexEditMarkupId === null) return null
  const veMarkup = pageMarkups.find(m => m.id === vertexEditMarkupId)
  if (!veMarkup || veMarkup.type === 'count') return null
  const preview = dragPreview?.type === 'vertex' && dragPreview.markupId === vertexEditMarkupId
    ? dragPreview
    : null
  const markupForHandles: Markup = preview
    ? { ...veMarkup, points: preview.points } as Markup
    : veMarkup
  return (
    <Layer listening={true}>
      <VertexHandleOverlay
        markup={markupForHandles}
        currentZoom={currentZoom}
        onHandleMouseDown={(vertexIndex) => {
          // Stub — wired in Plan 12-05 (vertex drag flow)
          void vertexIndex
        }}
      />
    </Layer>
  )
})()
```

Then in the JSX return, after the rubber-band layer:
```jsx
{vertexHandleLayer}
```

The named variable (rather than an inline IIFE in JSX) makes the `onHandleMouseDown` stub clearly visible and easy to replace in Plan 12-05.

**Why:** Keyboard handlers and the handle layer mount must be present before the drag flow plans (12-04, 12-05) build on them. The stub `onHandleMouseDown` ensures the overlay renders correctly for manual visual verification before the drag is wired.
