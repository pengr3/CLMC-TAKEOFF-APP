# Phase 03.1 Plan 04: Bug Gap Closure (B1 + B4) Summary

**Surgical fixes for B1 spacebar-hijack and B4 stale-zoom subscription — restores multi-word name entry and eliminates the "pins snap on next action" zoom glitch, both locked by regression tests.**

## Performance

## Accomplishments

- **B1 closed:** Spacebar keydown in useViewportControls now short-circuits when `isTextInputActive()` returns true. Users can type "Light Switch" and any multi-word name in MarkupNamePopup without the pan-mode side effect.
- **B4 closed:** `CanvasViewport.currentZoom` is now subscribed via `useViewerStore((s) => s.pageViewports[currentPage]?.zoom ?? 1)`. Zoom changes trigger a re-render, so every `X / currentZoom` calculation (calibration overlay, linear preview stroke, dependent label font sizes) refreshes in-step with the zoom gesture. The `?? 1` primitive fallback preserves Object.is stability — same pattern as the EMPTY_MARKUPS fix in commit `0e1a8e0`.
- **Regression tests:** 9 new tests across 2 files lock the fixes. The `viewport-subscription` suite proves the selector triggers a re-render on value change and skips re-renders when only unrelated viewport fields change. The `spacebar-text-guard` suite validates `isTextInputActive` for input/textarea/contenteditable/button/no-focus and includes a source-level check that the guard is applied before `preventDefault` in the hook.

## Decisions Made

- **Guard on keydown only, not keyup.** The keyup branch unconditionally clears `spaceHeld=false` — safe no-op when already false, and avoids the edge case where a user holds space, focuses an input (somehow), and releases: we still want to exit pan mode cleanly. No guard needed on keyup.
- **currentZoom declaration moved to top of component.** Previously it was declared inline just before the calibration-overlay constants. Moving it next to `currentPage`, `totalPages`, `getViewport`, `setViewport` groups all viewer-store subscriptions in one visually scannable block.
- **isTextInputActive attribute-fallback accepted as Rule 2.** jsdom does not reflect `el.contentEditable` IDL property. Production Chromium does, so runtime behavior is unchanged — but the helper is now more robust across test environments AND against legacy HTML that sets the attribute without touching the property. Three-line addition, zero risk.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing critical coverage] isTextInputActive contenteditable attribute fallback**
- **Found during:** Task 3 (regression tests)
- **Issue:** The contenteditable test case failed because jsdom's `setAttribute('contenteditable', 'true')` does not flip the `el.isContentEditable` IDL property or `el.contentEditable` setter return. Running the helper against a contenteditable element that was set via attribute-only returned `false`.
- **Fix:** Added an attribute-based fallback in `isTextInputActive` — reads `getAttribute('contenteditable')` and returns true for `''` or `'true'`. Matches HTML spec semantics (empty attribute value means contenteditable).
- **Files modified:** `src/renderer/src/hooks/useKeyboardShortcuts.ts`
- **Verification:** All 9 new tests pass; full suite (171 tests) passes; typecheck clean.
- **Committed in:** `9a34cc3` (Task 3 commit)

### Parallel-execution artifacts

**2. [Parallel-agent interaction] Task 1 commit bundled Plan 01's concurrent in-flight files**
- **Found during:** Task 1 commit inspection
- **Issue:** Plan 01 and Plan 04 ran in parallel. Between the `git status` snapshot and the `git commit` execution for Task 1, Plan 01's executor had modified `useMarkupTool.ts`, `markupStore.ts`, and two test files. These were included in my Task 1 commit even though only `useViewportControls.ts` was `git add`'d by name.

The resulting tree typechecks and tests pass cleanly. Plan 01's own commit `288bab8` followed immediately after, completing its plan.
- **Root cause:** Likely a race where Plan 01's editor saved files between my `git add` and `git commit` — the staging index picked them up. Non-destructive: both plans' intended changes landed.
- **Mitigation if repeated:** Use `git commit -o <file>` (only commit named files) or `git stash -u --keep-index` before commit. Noted for future parallel runs.

### Test adjustments (not plan deviations)

- **Source-level check used `path.resolve(process.cwd(), ...)` instead of `new URL(..., import.meta.url)`.** Vitest's `import.meta.url` is not always a `file://` URL on Windows, so `fs.readFileSync(new URL(...))` threw `TypeError: The URL must be of scheme file`. `path.resolve(process.cwd(), 'src/...')` is the portable equivalent.
- **Test files remain `.ts` (not `.tsx`).** vitest.config.ts restricts discovery to `src/tests/**/*.test.ts`. The plan noted this as a contingency; I used `React.createElement(...)` instead of JSX syntax to stay within the `.ts` extension.
