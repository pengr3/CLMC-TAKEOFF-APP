## Accomplishments

- **D-26 preset palette picker shipped** — MarkupNamePopup now renders 10 `role="radio"` swatch buttons inside a `role="radiogroup"` labelled "Markup color". Each swatch is a 24×24 rounded-square button showing the hex fill; the selected swatch has a 2px accent border + inset white ring so selection pops on the dark popup background.
- **D-25 inheritance working both ways** — on mount, the initial color resolves from `getColorForName(initialName)` if the name is known, otherwise falls back to `nextPaletteColor(usedColors)` scanning ALL pages. As the user types a name in the input, the effect re-runs and auto-switches the swatch when the name matches a stored markup.
- **Sticky user override** — once the user manually clicks a swatch, `userOverrodeColor.current = true` and the inheritance useEffect becomes a no-op for the remainder of the popup's life. Typing a known name after a manual click leaves the user's choice intact.
- **Payload widened to { name, categoryName, color }** — `handleConfirm` now threads `selectedColor` into the onConfirm call. `useMarkupTool.commitCountName` captures it into `pendingColor`; `recordClick`'s count branch stamps it into every pin. `commitShape` uses `payload.color` directly for linear/area/perimeter construction.
- **D-27 dead weight removed from CategoryAutocomplete** — the colored-dot span inside each dropdown item is gone. Category is now a pure name-grouping label. `itemStyle.gap` trimmed since there is only one child.
- **Temporary plan-01 bridge removed** — `resolveColorForName` (useMarkupTool.ts) is gone. Popup is now the single source of truth for markup color at placement time.
- **8 new render tests in markup-namepopup.test.ts** — lock every behavior above, including the less-obvious user-override-sticky contract that would be easy to accidentally regress.

## Decisions Made

- **Test file kept as .test.ts (not .test.tsx)** — vitest.config.ts's include glob is `src/tests/**/*.test.ts`. Changing it mid-wave risks conflicts with the parallel plan 02 executor. React.createElement is slightly more verbose than JSX but is the right call for 8 short focused tests.
- **Native value setter for controlled inputs in tests** — setting `input.value = 'x'` directly does NOT trigger React's onChange for controlled inputs in React 19 because React compares the current value against its prop-tracked value using the setter's intercepted invocation. The canonical technique is `Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value').set.call(input, 'x')` followed by dispatching a bubbling `input` event. This matches React Testing Library's internal approach.
- **userOverrodeColor is a useRef, not useState** — a ref is correct because (a) flipping it should not cause a re-render, and (b) the inheritance useEffect reads it without needing dependency-array inclusion. useState would create a stale-closure bug where the effect captures an old value of the flag.
- **initialColor prop added but not yet consumed at callsites** — plan 03.1-04 (context menu recolor) will pass initialColor when re-opening the popup for a known markup. Adding the prop surface here lets plan 04 stay focused on the context menu wiring.
- **CanvasViewport NOT touched** — the plan explicitly noted: "Unchanged at callsite — only the payload shape widens." TypeScript's default bivariance for method-style function-property parameters allowed `commitCountName: (p) => void` to satisfy `onConfirm: (pWider) => void` without cast. The commit discipline of this parallel wave was to avoid CanvasViewport edits from both plans simultaneously.
