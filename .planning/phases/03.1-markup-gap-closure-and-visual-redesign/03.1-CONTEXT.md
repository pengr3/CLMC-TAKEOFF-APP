<decisions>
## Implementation Decisions

### Count Pin Visual (supersedes UI-SPEC D-04 and D-13)

- **D-21:** Count pin = colored circle with the sequence number rendered INSIDE. No external "Name N" text on canvas.
- **D-22:** Pin size is **pure world-anchored** — no zoom compensation, no clamps. Pins scale 1:1 with the Stage transform, like a stamp on the drawing. Accepted trade-off: tiny at fit zoom, large at 8×. Matches CAD takeoff convention.
- **D-23:** Number ink inside the pin uses **auto-contrast** — compute perceived luminance of the fill color, pick white on dark fills and black on light fills. Works against any palette swatch.
- **D-24:** Item name is **never** on canvas. It appears only via hover tooltip (see D-30).
- **D-25:** Starting color for a newly placed count pin = **color of last-placed pin of the same name**. First pin in a new name-group picks next unused palette swatch.

### Color Model (supersedes MARK-08)

- **D-26:** Color picker in `MarkupNamePopup` = **preset palette only**, 8–12 visually distinct, accessibility-aware swatches. No hex input, no wheel. Palette set is fixed (Claude picks final swatches — accessible contrast, ~equally spaced on the color wheel).
- **D-27:** Category **retained** as a pure name-grouping label for BOQ export (Phase 5). Category never drives color. `CategoryAutocomplete` stays in the naming popup as a grouping-input aid; color swatches are removed from that dropdown.
- **D-28:** Color is picked in `MarkupNamePopup` **before** placing a markup. Post-placement recolor via **right-click context menu** on the pin/shape. Both paths write through the same store action so undo/redo works uniformly.
- **D-29:** Color is a property of the **name-group**, not of individual markups. Changing a color (anywhere — popup or context menu) for a given name retro-applies to every existing markup with that name. This mirrors the user's mental model: "Light Switches are blue" — one change flips them all. Must be a single undoable `MarkupCommand` (see Claude's Discretion below).

### Visual Identification

- **D-30:** Hover tooltip on **all four** markup types (count, linear, area, perimeter). Consistent UX. Tooltip shows at minimum the markup name; may include measurement when it helps (Claude's discretion).
- **D-31:** Measurement labels (e.g. "12.4 m", "8.3 m²") stay **visible on canvas** for linear/area/perimeter. Only the NAME moves to hover. Rationale: measurement IS the quantity data for a takeoff — hiding it would force constant hovering during verification.
- **D-32:** No legend panel in 3.1. Running-totals / live legend deferred to **Phase 6** (Live View). `README`-style mental model: "the canvas is the legend for now."
- **D-33:** Hover tooltip implementation = **HTML element, absolutely positioned** over the canvas area, driven by Konva `onMouseEnter`/`onMouseLeave` on each markup Group. Rationale: matches existing `MarkupNamePopup` / `ScalePopup` styling, escapes canvas bounds cleanly, no Stage re-render cost per hover.

### Label Legibility

- **D-34:** Measurement labels on linear/area/perimeter are **pure world-anchored at 14px** at 1× zoom (up from 12px). No screen-size clamps. Consistent with the count-pin "stamp on plan" metaphor in D-22. At fit zoom labels are tiny (user will zoom in to read quantities); at high zoom labels are large and punchy.
- **D-35:** Label background treatment — Claude's discretion. Default to drop-shadow/outline only (existing approach); prototype a semi-opaque rounded chip if shadow alone fails against dense construction drawings.

### Claude's Discretion

- **B1 fix** — `src/renderer/src/hooks/useViewportControls.ts` lines 130–136: guard the `e.preventDefault()` + `setSpaceHeld(true)` block with the existing `isTextInputActive()` helper exported from `useKeyboardShortcuts.ts`. Trivial 3-line change.
- **B2 fix (part A)** — `src/renderer/src/components/markup/LinearMarkup.tsx`: replace the vertex-index midpoint (`Math.floor(points.length / 2)`) with a true **arc-length midpoint** computed by walking `polylineLength`'s cumulative distance. Label lands on geometric center, not an arbitrary vertex.
- **B4 fix (foundational)** — `src/renderer/src/components/CanvasViewport.tsx` and the four markup components: subscribe to the viewport slice directly via `useViewerStore((s) => s.viewports[currentPage]?.zoom ?? 1)` (stable fallback, same pattern as the EMPTY_MARKUPS fix in commit `0e1a8e0`). Do NOT remove zoom-compensation from labels — only D-22 pins drop compensation. The calibration overlay and linear stroke widths still need fresh `currentZoom`.
- **MarkupCommand for color change** — new `RecolorGroup` command type covering the D-29 retro-apply. Stores `{name, oldColor, newColor}`; apply/undo iterate all markups matching the name. Keeps undo/redo parity with MARK-09.
- **Konva tooltip event wiring** — `Layer` containing committed markups needs `listening={true}` (currently off — see CanvasViewport line 429). Individual Group components attach `onMouseEnter` / `onMouseLeave` and emit pointer-space position up to a CanvasViewport-level tooltip renderer.
- **Palette selection** — 10 swatches from a curated accessible palette (leaning toward a variant of Tailwind's 500-level hues). Final list set during planning; planner should ensure contrast ratios meet WCAG AA against both white and black so auto-contrast in D-23 works.
- **Tooltip timing** — show after ~200ms hover (standard), hide on mouseleave. Follows native pattern; no fancy fade animations.

</decisions>

<specifics>
## Specific Ideas

- **"Pins as stamps on the plan"** — the core visual metaphor. Markups are part of the drawing, not floating overlay UI. Consequence: size scales with zoom by default (D-22, D-34), labels disappear at overview zoom (by design), details at work zoom.
- **"Barely readable" was a request for BIGGER, not for GONE** — user explicitly wants measurements visible on canvas (D-31). Count-pin text is the exception (user asked to remove it).
- **Color grouping by name, not by individual pin** — confirmed twice in discussion. Simpler than per-markup freedom, finer than per-category. "Light Switch is blue" holds for every Light Switch on the plan.
- **Hover tooltips are the name channel** — name is never on canvas. User trusts the discovery affordance (hover shows name) for reading, and the palette + color-per-name for visual grouping.

</specifics>
