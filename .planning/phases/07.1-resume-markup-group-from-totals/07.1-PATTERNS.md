```typescript
const handleArmTool = useCallback((item: BoqItemRow, categoryName: string): void => {
  const toolType = rowTypeToMarkupType(item.type)
  const itemName = labelToName(item.label)

  // Wall height: scan for most recent WallMarkup with this name
  // Pattern from markupStore.getColorForName() (markupStore.ts lines 225-237)
  let wallHeight: number | undefined
  if (toolType === 'wall') {
    const { pageMarkups } = useMarkupStore.getState()
    let latestWall: WallMarkup | null = null
    for (const list of Object.values(pageMarkups)) {
      for (const m of list) {
        if (m.type === 'wall' && m.name === itemName) {
          if (!latestWall || m.createdAt > latestWall.createdAt) {
            latestWall = m as WallMarkup
          }
        }
      }
    }
    wallHeight = latestWall?.wallHeight ?? 2400
  }

  // CRITICAL ORDERING: setChainArmedFromTotals BEFORE setActiveTool
  // Reason: activatePreset sets markupState.toolType so the sync effect at
  // CanvasViewport line 338-344 sees toolType===activeTool and skips activateMarkup()
  setChainArmedFromTotals({
    name: itemName,
    categoryName,
    color: item.color ?? MARKUP_PALETTE[0],
    toolType,
    wallHeight
  })
  useViewerStore.getState().setActiveTool(toolType)
}, [])
```

## Key Ordering Constraint

The single most important implementation rule for this phase, derived from RESEARCH.md Focus Area 2 / Pitfall 1:

In `App.tsx handleArmTool`, always call in this order:
1. `setChainArmedFromTotals(payload)` — writes `_chainArmedItem`, calls `activatePreset()` which sets `markupState.toolType = toolType`
2. `useViewerStore.getState().setActiveTool(toolType)` — triggers CanvasViewport sync effect, which checks `markupState.toolType !== activeTool` → FALSE (already set) → skips `activateMarkup()` override

Reversing the order causes the sync effect to call `activateMarkup()` which resets state to `INITIAL_STATE`, wiping the preset.
