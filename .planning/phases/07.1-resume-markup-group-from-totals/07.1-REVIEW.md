# Phase 7.1: Code Review Report

**Reviewed:** 2026-05-19
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

The Phase 7.1 implementation wires a "Resume Markup Group" flow from the TotalsPanel: clicking a BOQ row arms the matching markup tool with the row's name/category/color via `handleArmTool` in App.tsx, which calls `setChainArmedFromTotals` (writes to a module-level ref) then `setActiveTool` (triggers a React state change). The `activatePreset` function in `useMarkupTool` applies the preset atomically. Overall the architecture is sound.

Four blockers are present: a label-stripping regex divergence between `TotalsCategoryBlock` and `TotalsRow`/`App.tsx` that causes wall-type match failures; a race window where `setActiveTool` fires before `_activatePresetRef` is guaranteed to be populated in a fresh mount; the sync effect that activates the markup tool overwriting the preset state set by `activatePreset`; and a missing null guard in the wall-height scan that will throw when `pageMarkups` contains pages with no wall markups but the loop iterates over non-wall entries without type narrowing. Three warnings cover an `onRowClick` prop wired to a no-op in App.tsx (breaking the legacy click callback contract), a stale `cycleIndexByKey` prop never passed from App.tsx, and `getChainArmedItem()` being read as a side-effectful module ref inside JSX render without a re-render trigger when the ref changes independently of `activeTool`.

## Critical Issues

### CR-01: `labelToName` regex in `TotalsCategoryBlock` does not strip `wall` suffix — match failure for wall rows

**File:** `src/renderer/src/components/TotalsCategoryBlock.tsx:63`

**Issue:** `TotalsCategoryBlock.labelToName` strips only `(count|linear|area|perimeter)` from the label. `TotalsRow.labelToName` (the exported version used by App.tsx) additionally strips `wall`. For a row with label `"Wall (wall)"` the TotalsCategoryBlock helper returns `"Wall (wall)"` as the name, causing `matchesForRow` to find zero current-page matches and pass `[]` to `currentPageMatches`. The hover ring never lights up for wall rows, and `onArmTool` receives `currentPageMatches=[]` on hover.

The discrepancy also means `TotalsCategoryBlock.matchesForRow` will never match wall markups by name when the label carries the `(wall)` suffix, so PulseHighlight will not flash wall shapes after clicking their row.

**Fix:**
```typescript
// TotalsCategoryBlock.tsx line 63
function labelToName(label: string): string {
  return label.replace(/\s*\((count|linear|area|perimeter|wall)\)\s*$/, '')
}
```

### CR-02: `setActiveTool` triggers the sync effect BEFORE `activatePreset` can run — preset overwritten by plain `activate`

**File:** `src/renderer/src/components/CanvasViewport.tsx:368-374` and `src/renderer/src/App.tsx:260-267`

**Issue:** `handleArmTool` in App.tsx calls:
1. `setChainArmedFromTotals(payload)` — writes `_chainArmedItem` and synchronously calls `_activatePresetRef?.(...)` → `activatePreset(tool, preset)` → sets React state via `setState({...INITIAL_STATE, mode:'placing'|'drawing', chainArmed:true, pendingName, ...})`.
2. `useViewerStore.getState().setActiveTool(toolType)` — updates Zustand, which schedules a re-render of `CanvasViewport`.

On the very next render, the sync `useEffect` at line 368 runs:
```typescript
if (isMarkupTool(activeTool) && markupState.toolType !== activeTool) {
  activateMarkup(activeTool)   // calls plain activate(), not activatePreset()
}
```
Because `markupState.toolType` was just set to `toolType` by `activatePreset`, the condition `markupState.toolType !== activeTool` evaluates to **false** on the same render batch — but only if both state updates are batched. In React 18 automatic batching this is typically fine inside a single synchronous call. However, because `setChainArmedFromTotals` calls `activatePreset` (a hook callback captured in a React closure) synchronously from outside React's event system, the two updates are **not guaranteed to be batched**. If they flush in separate batches the effect fires with the new `activeTool` but the old `markupState.toolType === null` (still idle), causing `activateMarkup(activeTool)` to run and overwrite the preset state with `INITIAL_STATE` — erasing `pendingName`, `pendingCategoryName`, and `chainArmed: true`.

Additionally, when `CanvasViewport` is unmounted and remounted (e.g. switching from `EmptyState` to `CanvasViewport` when the first PDF loads mid-session), `_activatePresetRef` is set to `null` in the cleanup of the `useEffect` at line 643-647. If `setChainArmedFromTotals` is called in the small window between unmount cleanup and the next `useEffect` registration, `_activatePresetRef` is null and `activatePreset` silently does not fire at all — leaving the tool armed in `viewerStore` but with no preset state in the markup machine.

**Fix:** Guard the sync effect so it does not call `activate` when the tool state machine is already in a non-idle mode for the correct tool (i.e., `activatePreset` already ran):
```typescript
useEffect(() => {
  if (isMarkupTool(activeTool) && markupState.toolType !== activeTool) {
    // Only call plain activate if the machine is idle or on a different tool.
    // activatePreset sets toolType + mode atomically; skip if already correct type
    // and mode is not idle (preset already applied).
    if (markupState.mode === 'idle') {
      activateMarkup(activeTool)
    }
  } else if (!isMarkupTool(activeTool) && markupState.mode !== 'idle') {
    cancelMarkup()
  }
}, [activeTool, markupState.toolType, markupState.mode, activateMarkup, cancelMarkup])
```
This stops the effect from clobbering an already-armed preset.

### CR-03: Wall-height scan iterates all pages but never type-narrows — `latestWall` comparison is unsafe

**File:** `src/renderer/src/App.tsx:245-256`

**Issue:** The wall-height scan loop:
```typescript
for (const markups of Object.values(pageMarkups)) {
  for (const m of markups) {
    if (m.type === 'wall' && m.name === itemName) {
      if (latestWall === null || m.createdAt > latestWall.createdAt) {
        latestWall = m as WallMarkup
      }
    }
  }
}
wallHeight = latestWall?.wallHeight ?? 2400
```
The cast `m as WallMarkup` is correct only inside the `m.type === 'wall'` branch, and TypeScript accepts the narrowing. However, the fallback `wallHeight = latestWall?.wallHeight ?? 2400` only runs when `toolType === 'wall'` — this part is safe.

The real bug is that `Object.values(pageMarkups)` returns `Markup[][]` (arrays per page), but in the store schema `pageMarkups` is typed as `Record<number, Markup[]>`. If the store ever contains a page key that maps to `undefined` (possible if a page was deleted without cleaning up the record), `Object.values` will include `undefined` in the iteration and `for (const m of markups)` will throw `TypeError: undefined is not iterable`. There is no null guard on the inner array.

**Fix:**
```typescript
for (const markups of Object.values(pageMarkups)) {
  if (!markups) continue   // guard against sparse pages
  for (const m of markups) {
```

### CR-04: `handleArmTool` in App.tsx uses `rowTypeToMarkupType` which maps `perimeter-length`/`perimeter-area` to `'perimeter'` — but `setActiveTool('perimeter')` never activates a `wall` tool for wall rows

**File:** `src/renderer/src/App.tsx:239`

**Issue:** This is correct for perimeter rows, but the `BoqRowType` for a wall row is `'wall'`, and `rowTypeToMarkupType('wall')` returns `'wall'` (the default branch). This is fine. The bug is subtler: `TotalsRow.handleClick` calls `onArmTool?.(item)` where `item.type` is the `BoqRowType`. `TotalsCategoryBlock` wraps this as `onArmTool ? (it) => onArmTool(it, category.name) : undefined`. The `categoryName` passed to `handleArmTool` is `category.name` from the `TotalsCategoryBlock`. However, `TotalsPanel` passes `onArmTool` directly without providing a `categoryName` default — if `onArmTool` is not provided, the prop is `undefined` and TotalsRow silently skips arm. This is all correct.

The actual blocker: when `item.color` is `null` or `undefined` (which is valid per the type — `BoqItemRow.color` is `string | undefined`), the fallback `item.color ?? MARKUP_PALETTE[0]` supplies the first palette color. But the `MARKUP_PALETTE[0]` import in App.tsx at line 27 is only used here. If `MARKUP_PALETTE` is an empty array (which would be a programming error in markup-palette.ts but is not guarded against), `MARKUP_PALETTE[0]` is `undefined`, and `setChainArmedFromTotals` receives `color: undefined`, which TypeScript should reject — but the type signature of `setChainArmedFromTotals` declares `color: string` (non-optional), so this silently passes `undefined` as a `string` at runtime if the palette is empty.

More concretely: there is no guard that `MARKUP_PALETTE.length > 0` before indexing. This is a latent crash risk.

**Fix:**
```typescript
color: item.color ?? (MARKUP_PALETTE[0] ?? '#1677ff'),
```

### WR-01: `onRowClick` is never wired in App.tsx — `TotalsPanel` receives no `onRowClick` prop

**File:** `src/renderer/src/App.tsx:402-412`

**Issue:** `TotalsPanel` is rendered at line 402 without an `onRowClick` prop. `TotalsPanel` falls back to `handleRowClick = onRowClick ?? (() => {})` — a permanent no-op. `TotalsRow.handleClick` calls `onClick(item)` (the legacy observer), which is always the no-op. Any caller in App.tsx that depended on `onRowClick` to observe row clicks (e.g. for analytics, selection sync, or future integrations) will silently receive nothing. This is a prop contract gap — the interface declares the prop optional, but given it was purposely included in the interface it likely has an expected downstream consumer.

**Fix:** Wire a handler or explicitly document that `onRowClick` is intentionally unused in this phase:
```typescript
// Either wire it:
onRowClick={(item, _cat) => { /* e.g. log or future use */ }}
// Or document the omission in a comment above the TotalsPanel render.
```

### WR-02: `cycleIndexByKey` is never passed from App.tsx — `TotalsRow` always receives `cycleIndex={0}`

**File:** `src/renderer/src/App.tsx:402-412` and `src/renderer/src/components/TotalsPanel.tsx:255`

**Issue:** `TotalsPanel` accepts `cycleIndexByKey?: Record<string, number>` and threads it to `TotalsCategoryBlock`. App.tsx never passes this prop, so it is always `undefined`. `TotalsCategoryBlock` then computes `cycleIndexByKey?.[key] ?? 0` → always 0. The `cycleIndexProp` in `TotalsRow` is always 0.

`TotalsRow` maintains its own `internalCycleIndex` (starting at 0) to drive `showCycleDot`, so the cycle-dot feature itself works. But the parent-driven cycle prop path (`cycleIndexProp > 0`) is permanently dead — any code that expected the parent to drive the cycle dot will never fire. If this prop was intended to be wired in a future wave, the dead code path creates confusion.

**Fix:** Either wire `cycleIndexByKey` from App.tsx (if needed) or remove the prop from the interface and `TotalsCategoryBlock`/`TotalsRow` until it has a concrete use.

### WR-03: `getChainArmedItem()` read in JSX render is a module-ref side-read — badge does not re-render when ref changes independently

**File:** `src/renderer/src/components/RibbonToolbar.tsx:397-399, 409-411, 421-423, 433-435, 445-447`

**Issue:** `RibbonToolbar` reads `getChainArmedItem()` inside JSX during render:
```tsx
{activeTool === 'count' && getChainArmedItem() !== null && (
  <ChainBadge color={getChainArmedItem()!.color} name={getChainArmedItem()!.name} />
)}
```
`getChainArmedItem()` returns `_chainArmedItem`, a module-level variable. React does not subscribe to module-level variable changes. The badge renders correctly only because the `activeTool` state change (which triggers a `RibbonToolbar` re-render) happens in the same event as the ref write. However there is a narrow window:

1. `setChainArmedFromTotals` writes `_chainArmedItem` (sync).
2. `_activatePresetRef?.()` fires `activatePreset` (sync setState, batched).
3. `setActiveTool` writes Zustand (sync, batched).

If `setActiveTool` triggers `RibbonToolbar` re-render before `CanvasViewport`'s `useEffect` at line 651-654 runs (which writes `_chainArmedItem` from React state back to the module ref), the badge may briefly show stale data. More importantly: if the user cancels the tool (Escape → `cancelMarkup` → `setActiveTool('select')`), `_chainArmedItem` is cleared by `CanvasViewport`'s effect on next render — but `RibbonToolbar` only re-renders when `activeTool` changes, which it does (to `'select'`), so the badge disappears correctly. The race window exists but the visible impact is limited to a single-frame flicker.

The deeper issue is that calling `getChainArmedItem()` **twice** in the same JSX expression (null check + property reads) means if the ref changes between those two reads (theoretically impossible in synchronous render, but a code-smell): `getChainArmedItem()!.color` and `getChainArmedItem()!.name` could see stale data.

**Fix:** Capture once per render:
```tsx
const chainArmed = getChainArmedItem()
// then use `chainArmed` throughout the JSX
{activeTool === 'count' && chainArmed !== null && (
  <ChainBadge color={chainArmed.color} name={chainArmed.name} />
)}
```
