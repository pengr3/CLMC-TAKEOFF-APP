## Interaction Contract

This section documents the precise interaction states, triggers, and visual changes introduced by Phase 7.1. It is the primary specification for the executor.

### 7.1-I-01: Row Hover Affordance — "+" Icon

**Trigger:** `onMouseEnter` on a `TotalsRow` that has a resolved `item.type` (all rows qualify).

**Visual change:** A "+" icon (lucide-react `Plus`, size=14, `color={COLORS.textSecondary}`) appears in a new fixed-width 20px slot appended to the right end of the row, after the UoM column. The slot is always present in the DOM but the icon is conditionally rendered (`hovered ? <Plus/> : null`) — the 20px width is held regardless, so the UoM column does not reflow.

**Visual non-change:** The existing hover background (`COLORS.hoverSurface`) continues to apply as before. No additional background change for the "+" slot.

**On leave:** `onMouseLeave` — the "+" icon disappears. Slot remains 20px wide.

**Interaction gate:** The "+" is a passive visual affordance only. It has no separate click handler. The row's existing `onClick` handler (which fires on the full row click) is where the arm action lives.

### 7.1-I-02: Row Click — Arm Tool

**Trigger:** `onClick` on any `TotalsRow` (left mouse click, not right-click).

**Action sequence:**
1. Resolve the `MarkupToolType` from the row's `item.type` via `rowTypeToMarkupType()` (existing function — `perimeter-length` and `perimeter-area` both map to `'perimeter'`).
2. Derive the `itemName` from `labelToName(item.label)` (existing function — strips disambiguation suffix).
3. Resolve `categoryName` from the markup store: look up the category whose `id === item.categoryId`. If not found, use `'Uncategorized'`.
4. Use `item.color` as the armed color.
5. Call `useViewerStore.getState().setActiveTool(resolvedMarkupType)` to arm the tool in the ribbon.
6. Call `setChainArmedFromTotals({ name: itemName, categoryName, color: item.color, toolType: resolvedMarkupType })` — a new module-level setter on CanvasViewport (mirrors `_chainArmedItem` pattern, see §Component Contracts below).

**Result:** The ribbon's Tools tab shows the named tool as active (blue underline). The ChainBadge chip appears on the tool button (because `getChainArmedItem()` is now non-null and `activeTool === toolType`). The canvas cursor switches to the CROSSHAIR_CURSOR (existing behavior — triggered by `isMarkupTool(activeTool) === true`).

**Existing cycle navigation:** The existing page-cycle behavior (navigating to the next page with matches) continues to fire on click. The arm action fires AFTER the cycle navigation — both happen in the same `onClick` handler. The arm does not suppress or replace cycle navigation.

**Wall markup special case:** When `item.type === 'wall'`, the armed item must also carry the existing `pendingWallHeight`. Resolution order:
1. Read the most recent WallMarkup with `name === itemName` from the markup store and use its `wallHeight` field.
2. If no WallMarkup found with that name, default to `2400` (the `INITIAL_STATE.pendingWallHeight` constant).
This prevents the user from being prompted for wall height again when resuming an existing wall group.

### 7.1-I-03: Armed State Visual — Chain Badge on Ribbon

**No new work required.** The existing `ChainBadge` component in `RibbonToolbar.tsx` reads `getChainArmedItem()` at render time. Once `setChainArmedFromTotals()` populates the module-level `_chainArmedItem` ref, the next ribbon render (driven by `activeTool` change) will display the badge automatically.

The badge shows: 8px colored circle + item name (truncated at 64px max-width, ellipsis). Background: `COLORS.activeSurface`. Font: 10px, weight 600.

### 7.1-I-04: Disarm — Esc or Second Click on Armed Tool Button

**Trigger A (Esc):** The existing `cancel()` call in `useMarkupTool` resets `chainArmed` to false and calls `INITIAL_STATE` reset. The `_chainArmedItem` ref on CanvasViewport must be set to `null` by the same code path that calls `cancel()`. No new escape handler needed — wire `setChainArmedFromTotals(null)` into the existing cancel paths.

**Trigger B (second click on tool button):** The existing `handleMarkupToolClick` in `RibbonToolbar` calls `setActiveTool('select')` when `activeTool === tool`. This already disarms the ribbon visually. The `_chainArmedItem` ref must also be cleared here — wire `setChainArmedFromTotals(null)` inside `handleMarkupToolClick` when the condition `activeTool === tool` is true.

**Visual result:** ChainBadge disappears. Canvas cursor reverts to default. Tool button loses active state.

### 7.1-I-05: Chain Mode Continues After Arming from Totals

**Requirement SC-4:** After arming from the totals panel, successive placements use the same name/category/color without re-prompting. This is the existing chain mode behavior — it is fully automatic because `commitShape` and `commitCountName` already set `chainArmed: true` on first commit and preserve `pendingName`/`pendingCategoryName`/`pendingColor`. No additional logic is needed for chain continuation once the initial arm is set up correctly.

**Count tool specifics:** For count, the tool enters `mode: 'naming'` on `activate()`. When arming from totals, the tool should skip the naming popup and jump directly to `mode: 'placing'` with the pre-filled name/category/color. This requires a new `activateWithPreset(payload)` path or equivalent — see §Component Contracts below.

### 7.1-I-06: Wall Height Inheritance — No Popup on Resume

When the armed tool type is `'wall'` and the wall height was resolved from an existing WallMarkup (7.1-I-02 step), the MarkupNamePopup wall-height row should pre-fill with the resolved height when it appears at commit time. The user is not re-prompted before drawing begins — wall height only appears in the commit popup (existing behavior).

## Component Contracts

### TotalsRow — Changes

**New slot:** Add a 20px wide, flexShrink=0 slot after the UoM `<span>`. The slot holds a `<Plus size={14} color={COLORS.textSecondary} />` icon rendered only when the row is hovered (`bg !== 'transparent'`). The slot is present in the DOM at all times to prevent reflow.

**New prop:** `onArmTool?: (item: BoqItemRow, categoryName: string) => void` — fired from `handleClick` after the existing cycle-navigation logic. The parent resolves category name and coordinates the arm action.

**Existing props:** All existing props are unchanged. `onClick` remains for legacy observers.

**Hover state reuse:** The existing `bg` state variable already tracks hover (`COLORS.hoverSurface` on enter, `'transparent'` on leave). Drive the "+" visibility from `bg !== 'transparent'`.

### TotalsCategoryBlock — Changes

**New prop:** `onArmTool?: (item: BoqItemRow, categoryName: string) => void` — threaded from TotalsPanel → TotalsCategoryBlock → TotalsRow. The `categoryName` is already available as `category.name` inside the block.

**Wire:** In the `TotalsRow` render inside `TotalsCategoryBlock`, pass `onArmTool={(it) => onArmTool?.(it, category.name)}`.

### TotalsPanel — Changes

**New prop:** `onArmTool?: (item: BoqItemRow, categoryName: string) => void` — accepted and passed to `TotalsCategoryBlock`.

**No other panel-level changes.**

### CanvasViewport — Changes (module-level ref pattern)

Following the exact pattern of `_chainArmedItem` / `getChainArmedItem()` established in Phase 8:

```
// New exported setter (mirrors _chainArmedItem write path)
export function setChainArmedFromTotals(
  payload: { name: string; categoryName: string; color: string; toolType: MarkupToolType; wallHeight?: number } | null
): void
```

This setter writes to the existing `_chainArmedItem` ref AND additionally arms the `useMarkupTool` state machine with the correct preset (bypassing the naming popup).

**activateWithPreset path for count tool:** The `useMarkupTool.activate()` currently always opens the naming popup for count. A new `activateWithPreset()` exported from the hook (or a parameter on `activate()`) must put the count tool directly into `mode: 'placing'` with the provided name/category/color. Suggested signature addition to `UseMarkupToolReturn`:

```
activatePreset: (tool: MarkupToolType, preset: { name: string; categoryName: string; color: string; wallHeight?: number }) => void
```

This method:
- For `count`: sets `mode: 'placing'`, `toolType: 'count'`, `pendingName`, `pendingCategoryName`, `pendingColor`, `chainArmed: true`
- For `linear` / `area` / `perimeter` / `wall`: sets `mode: 'drawing'`, `toolType`, pending fields, `chainArmed: true`, and for wall: `pendingWallHeight`

### App.tsx — Changes

**New handler:** `handleArmTool(item: BoqItemRow, categoryName: string)` — called via `onArmTool` prop on TotalsPanel. This handler:
1. Resolves `toolType` from `rowTypeToMarkupType(item.type)`.
2. Resolves `wallHeight` for wall items (scan markup store).
3. Calls `setActiveTool(toolType)` on viewerStore.
4. Calls `setChainArmedFromTotals(payload)` exported from CanvasViewport.

**Wire:** Pass `onArmTool={handleArmTool}` to `<TotalsPanel>`.

## State Machine Contract

The arm-from-totals flow maps onto the existing `MarkupDrawState` in `useMarkupTool.ts` as follows:

```
[idle] ──── armFromTotals(count) ────→ [placing] (chainArmed=true, pre-filled)
[idle] ──── armFromTotals(linear)────→ [drawing] (chainArmed=true, pre-filled)
[idle] ──── armFromTotals(area)  ────→ [drawing] (chainArmed=true, pre-filled)
[idle] ──── armFromTotals(perim) ────→ [drawing] (chainArmed=true, pre-filled)
[idle] ──── armFromTotals(wall)  ────→ [drawing] (chainArmed=true, pre-filled, pendingWallHeight set)

[placing|drawing] ──── Esc ──────────→ [idle]   (chainArmed=false, _chainArmedItem=null)
[placing|drawing] ──── re-click tool ─→ [idle]   (chainArmed=false, _chainArmedItem=null)
```

The arm does NOT skip through `'naming'` or `'confirming'` — it bypasses them entirely and lands directly in the correct placement/drawing mode.
