# Phase 7.1: Resume Markup Group from Totals Panel - Research

**Researched:** 2026-05-19
**Domain:** React component prop threading, module-level ref pattern, useMarkupTool state machine extension
**Confidence:** HIGH

---

## Summary

Phase 7.1 adds a single interaction: clicking a TotalsRow arms the matching markup tool with that item's name, category, and color. The feature slots into a well-defined architecture that already handles chain mode, module-level ref communication between CanvasViewport and the RibbonToolbar, and a state machine (`useMarkupTool`) that has clearly delineated mode transitions. No new libraries, IPC channels, or stores are needed.

The core work is three surgical changes: (1) add `activatePreset()` to `useMarkupTool` so the state machine can skip the naming popup and land directly in `mode:'placing'` (count) or `mode:'drawing'` (all others) with pre-filled pending fields; (2) add `setChainArmedFromTotals()` as an exported setter on `CanvasViewport` that writes the existing `_chainArmedItem` ref AND calls `activatePreset()`; (3) thread `onArmTool` from `App.tsx` → `TotalsPanel` → `TotalsCategoryBlock` → `TotalsRow` and add the `+` hover affordance on each row.

The critical interaction subtlety: `cancel()` in `useMarkupTool` already resets everything to `INITIAL_STATE` (which sets `chainArmed: false`). The only gap is that the `_chainArmedItem` ref must be set to `null` via `setChainArmedFromTotals(null)` in two places: the Esc path that calls `cancelMarkup()`, and the `handleMarkupToolClick` re-click path in `RibbonToolbar`.

**Primary recommendation:** The `activatePreset()` path is a single new callback added to `UseMarkupToolReturn` — it does not modify any existing code path and is safe to implement in isolation before threading the UI.

### Focus Area 1: TotalsRow Click/Hover Architecture

**File:** `src/renderer/src/components/TotalsRow.tsx`

The existing `handleClick()` (line 153) does two things:
1. Calls `onClick(item)` — the legacy callback (line 155)
2. Runs cycle navigation: reads `pagesWithMatches`, calls `setPage(targetPage)`, fires `onTriggerPulse?.(...)`, advances `internalCycleIndex` (lines 157–172)

The cycle navigation short-circuits with `return` at line 158 if `pagesWithMatches.length === 0`. The arm action must be appended AFTER the cycle navigation — even if cycle returns early, the arm should still fire (an item with no matching pages on any page can still be armed for new placements).

The hover state is tracked via `bg` useState (lines 105, 137–146):
- `onMouseEnter`: `setBg(COLORS.hoverSurface)`
- `onMouseLeave`: `setBg('transparent')`
- The UI spec drives `+` icon visibility from `bg !== 'transparent'` — this is exactly the existing variable.

**New prop required on TotalsRow:** `onArmTool?: (item: BoqItemRow, categoryName: string) => void`

The `labelToName()` helper (lines 62–64) strips `\s*\((count|linear|area|perimeter|wall)\)\s*$` suffix. This gives the clean name to pass to the arm action.

The `rowTypeToMarkupType()` helper (lines 67–70) maps `BoqRowType` → `MarkupType`:
- `'perimeter-length'` → `'perimeter'`
- `'perimeter-area'` → `'perimeter'`
- all others pass through unchanged (`'count'`→`'count'`, `'linear'`→`'linear'`, `'area'`→`'area'`, `'wall'`→`'wall'`)

Note: `rowTypeToMarkupType` is duplicated in both `TotalsRow.tsx` (line 67) and `TotalsCategoryBlock.tsx` (line 46). The planner can import from either location or App.tsx can call it from `TotalsRow.tsx` if exported.

### Focus Area 2: _chainArmedItem Module-Ref Pattern

**File:** `src/renderer/src/components/CanvasViewport.tsx`

The pattern is fully established at lines 124–128:
```
let _chainArmedItem: { name: string; color: string } | null = null
export function getChainArmedItem(): { name: string; color: string } | null {
  return _chainArmedItem
}
```

The ref is WRITTEN inside a `useEffect` at lines 612–616:
```typescript
useEffect(() => {
  _chainArmedItem = (markupState.chainArmed && markupState.pendingName)
    ? { name: markupState.pendingName, color: markupState.pendingColor }
    : null
}, [markupState.chainArmed, markupState.pendingName, markupState.pendingColor])
```

**`setChainArmedFromTotals()` does NOT exist yet.** It must be added as a new exported function. It is NOT a useEffect-written setter — it must be an imperative module-level setter callable from outside React.

The new setter will need to:
1. Write `_chainArmedItem` directly (same pattern as `_canvasControls` setter at line 593)
2. Call `activatePreset()` on the hook instance — but the hook is inside the component. This means `setChainArmedFromTotals` cannot call the hook directly; instead it must reach the hook via another module-level ref (following the same pattern as `_calibrationControls` at lines 113–121 and its useEffect at lines 600–609).

**The correct implementation:** Add a module-level ref for the `activatePreset` function, populated via `useEffect` (mirroring `_calibrationControls`). Then `setChainArmedFromTotals()` calls this ref AND sets `_chainArmedItem`.

The existing sync effect at CanvasViewport line 338–344 watches `activeTool` and calls `activateMarkup(activeTool)` when `activeTool` changes. This effect MUST be guarded so it does not override the preset state set by `activatePreset()`. The cleanest approach: `activatePreset()` sets a flag in the hook state (or the existing `chainArmed: true` field acts as the guard), and the sync effect checks `markupState.chainArmed` before calling `activateMarkup`.

**Risk:** The sync effect at line 338–344 is:
```typescript
useEffect(() => {
  if (isMarkupTool(activeTool) && markupState.toolType !== activeTool) {
    activateMarkup(activeTool)  // <-- this would OVERRIDE the preset
  } else if (!isMarkupTool(activeTool) && markupState.mode !== 'idle') {
    cancelMarkup()
  }
}, [activeTool, markupState.toolType, markupState.mode, activateMarkup, cancelMarkup])
```

When `setChainArmedFromTotals` calls `setActiveTool(toolType)` AND `activatePreset()`, the effect will fire because `activeTool` changed. If `activatePreset()` sets `toolType` to match the new `activeTool`, the condition `markupState.toolType !== activeTool` will be false and `activateMarkup()` will NOT fire. This is the correct ordering: `activatePreset()` must set `toolType` in the state so the sync effect sees a match and skips the override.

### Focus Area 7: ChainBadge Rendering Trigger

**File:** `src/renderer/src/components/RibbonToolbar.tsx` (lines 86–126, 396–447)

`ChainBadge` is a local function component (line 86). It renders an 8px color circle + truncated name string.

The rendering condition for each tool button (example for Count, lines 396–398):
```typescript
{activeTool === 'count' && getChainArmedItem() !== null && (
  <ChainBadge color={getChainArmedItem()!.color} name={getChainArmedItem()!.name} />
)}
```

`getChainArmedItem()` is called DIRECTLY during render — not via a Zustand selector. This means the badge will appear as soon as the component re-renders. Re-renders are driven by the `activeTool` Zustand subscription (line 136: `const activeTool = useViewerStore((s) => s.activeTool)`). When `setActiveTool(toolType)` is called from App.tsx's `handleArmTool`, the `activeTool` change propagates to `RibbonToolbar`, which re-renders and reads the updated `_chainArmedItem` ref.

**Sequence matters:** `_chainArmedItem` must be set BEFORE `setActiveTool()` fires so the re-render reads the non-null value. In practice, since `setChainArmedFromTotals()` is synchronous and the `activeTool` change is also synchronous (Zustand setState), the ordering within `handleArmTool` is:
1. Set `_chainArmedItem` (via `setChainArmedFromTotals`)
2. Call `setActiveTool(toolType)` (triggers re-render)

This ensures the badge appears on the same render as the tool activation.

The `_chainArmedItem` type is currently `{ name: string; color: string } | null`. The UI spec does NOT require the type to carry `toolType` — the badge only shows `name` and `color`. No type change needed.

### Focus Area 8: RibbonToolbar handleMarkupToolClick

**File:** `src/renderer/src/components/RibbonToolbar.tsx` (lines 193–201)

```typescript
const handleMarkupToolClick = (
  tool: 'count' | 'linear' | 'area' | 'perimeter' | 'wall'
): void => {
  if (activeTool === tool) {
    setActiveTool('select')   // disarm path — also need setChainArmedFromTotals(null) here
  } else {
    setActiveTool(tool)
  }
}
```

When `activeTool === tool` (re-click disarms): the `_chainArmedItem` ref must be cleared. Options:
1. Import `setChainArmedFromTotals` from `CanvasViewport` and call it inside `handleMarkupToolClick` when the disarm branch fires.
2. The CanvasViewport effect at lines 612–616 already sets `_chainArmedItem = null` when `markupState.chainArmed === false`. When `setActiveTool('select')` fires, CanvasViewport's sync effect calls `cancelMarkup()`, which sets `INITIAL_STATE` (chainArmed=false), which triggers the effect → sets `_chainArmedItem = null`.

**Observation:** The existing effect chain (activeTool change → sync effect → cancelMarkup → chainArmed=false → chainArmed effect → _chainArmedItem=null) already clears the ref automatically when the tool is disarmed via re-click. So for the re-click path, no additional wire may be needed. However, the `setChainArmedFromTotals` module-level ref function also needs to be set to null immediately for any imperative callers outside the React lifecycle. The safest approach: call `setChainArmedFromTotals(null)` explicitly in the disarm branch to be defensive, in addition to the effect chain.

**File:** `src/renderer/src/hooks/useMarkupTool.ts` (lines 115–118)


## Architecture Patterns

### Data Flow: Row Click → Tool Armed

```
TotalsRow.handleClick()
  ↓ existing: cycle navigation
  ↓ new: calls onArmTool(item, categoryName)
  
TotalsCategoryBlock
  ↓ threads onArmTool=(it) => onArmTool(it, category.name)
  
TotalsPanel
  ↓ threads onArmTool

App.tsx handleArmTool(item, categoryName)
  1. rowTypeToMarkupType(item.type) → toolType
  2. labelToName(item.label) → itemName
  3. item.categoryId → getCategory(categoryId) → categoryName (or use passed categoryName)
  4. item.color → armedColor
  5. If toolType === 'wall': scan markupStore for latestWallHeight
  6. setChainArmedFromTotals({ name, categoryName, color, toolType, wallHeight? })
     → writes _chainArmedItem
     → calls _activatePresetRef.current(toolType, preset)
        → useMarkupTool.activatePreset() sets state to placing/drawing + chainArmed=true
  7. viewerStore.setActiveTool(toolType)
     → triggers RibbonToolbar re-render → reads _chainArmedItem → shows ChainBadge
     → triggers CanvasViewport sync effect → sees markupState.toolType === activeTool → skips activateMarkup()
```

### setChainArmedFromTotals Implementation Pattern

Following `_calibrationControls` (CanvasViewport lines 113–121 + lines 600–609):

```typescript
// NEW module-level ref — mirrors _calibrationControls pattern
type ActivatePresetFn = (
  tool: MarkupToolType,
  preset: { name: string; categoryName: string; color: string; wallHeight?: number }
) => void
let _activatePresetRef: ActivatePresetFn | null = null

// NEW exported setter — called from App.tsx handleArmTool
export function setChainArmedFromTotals(
  payload: { name: string; categoryName: string; color: string; toolType: MarkupToolType; wallHeight?: number } | null
): void {
  if (payload === null) {
    _chainArmedItem = null
    return
  }
  _chainArmedItem = { name: payload.name, color: payload.color }
  _activatePresetRef?.(payload.toolType, {
    name: payload.name,
    categoryName: payload.categoryName,
    color: payload.color,
    wallHeight: payload.wallHeight
  })
}

// Inside CanvasViewport component — NEW useEffect
useEffect(() => {
  _activatePresetRef = activatePreset  // from useMarkupTool return
  return () => { _activatePresetRef = null }
}, [activatePreset])
```

### Sync Effect Guard (CanvasViewport lines 338–344)

After `setChainArmedFromTotals` is called with a payload, `activatePreset()` fires synchronously (same JS call stack tick) setting `markupState.toolType = toolType`. Then `setActiveTool(toolType)` is called, changing `activeTool`. The sync effect fires and checks `markupState.toolType !== activeTool` — since `activatePreset` already set `toolType`, this is FALSE, so `activateMarkup()` is NOT called. The preset is preserved.

**Ordering requirement in App.tsx handleArmTool:**
```
1. setChainArmedFromTotals(payload)   // sets _chainArmedItem + calls activatePreset
2. viewerStore.setActiveTool(toolType)  // triggers re-renders
```
NOT reversed. If `setActiveTool` fires first, the sync effect sees `toolType !== activeTool` and calls `activateMarkup()`, overriding the preset.

## Common Pitfalls

### Pitfall 1: Sync Effect Override
**What goes wrong:** `setActiveTool(toolType)` fires before `activatePreset()`, causing the sync effect at CanvasViewport line 338–344 to call `activateMarkup(tool)` which resets state to `INITIAL_STATE` — wiping the preset.
**Why it happens:** The sync effect runs on every `activeTool` change. If `markupState.toolType` doesn't yet match the new `activeTool`, `activateMarkup` fires.
**How to avoid:** Always call `setChainArmedFromTotals(payload)` (which internally calls `activatePreset`) BEFORE `setActiveTool(toolType)`.
**Warning signs:** ChainBadge appears but mode stays `'naming'` (count) or `'idle'` instead of `'placing'`/`'drawing'`.

### Pitfall 2: _activatePresetRef is null at call time
**What goes wrong:** `setChainArmedFromTotals()` calls `_activatePresetRef?.()` but the ref is null because CanvasViewport hasn't mounted yet (no PDF open, or component unmounted).
**Why it happens:** The `useEffect` that populates `_activatePresetRef` only runs after CanvasViewport mounts. If `handleArmTool` is called before mount (impossible in normal flow, but defensive programming matters), the ref is null.
**How to avoid:** The optional chain `_activatePresetRef?.()` already handles null gracefully. No markup tool button is enabled when `totalPages === 0`, so this path can't be triggered in practice.

### Pitfall 3: Double render of ChainBadge
**What goes wrong:** `_chainArmedItem` is set, then the CanvasViewport effect at lines 612–616 runs and overwrites it based on `markupState.chainArmed`. If `activatePreset` set `chainArmed: true` correctly, the effect will compute `_chainArmedItem = { name, color }` — same value — so no visible bug. But if `pendingName` is empty in the state, the effect sets `_chainArmedItem = null`.
**Why it happens:** The effect at lines 612–616 is the canonical writer for `_chainArmedItem`. After `activatePreset` sets `pendingName`, `pendingColor`, and `chainArmed: true`, the next render fires this effect with the correct values. The badge will survive.
**How to avoid:** Ensure `activatePreset` always sets non-empty `pendingName` from the arm payload. Empty name should be rejected upstream (only armed from existing items which always have names).

### Pitfall 4: category name vs categoryId confusion
**What goes wrong:** `BoqItemRow.categoryId` is the UUID, not the name. `TotalsCategoryBlock.category.name` is the display name. The arm action needs the display name (as a string) to pass to `useMarkupTool.activatePreset()`, which passes it to `store.getOrCreateCategory(categoryName)`.
**Why it happens:** The threaded `onArmTool(item, categoryName)` passes `category.name` from `TotalsCategoryBlock` — this is correct. App.tsx should use `categoryName` (the second argument), NOT try to look up from `item.categoryId`.
**How to avoid:** The `categoryName` argument in `handleArmTool(item: BoqItemRow, categoryName: string)` already carries the display string. Use it directly.

### Pitfall 5: onArmTool fires even when arm should be suppressed
**What goes wrong:** A lightbulb click (TotalsRow line 235 with `e.stopPropagation()`) already prevents click from propagating to the row. But if `onArmTool` is called from `handleClick`, it fires on every row click including ones the user intended as cycle-only navigation.
**Why it happens:** Per the UI spec (7.1-I-02), clicking ANY row arms the tool. There is no suppression for the arm. This is intentional — the arm always fires alongside cycle navigation.
**How to avoid:** This is by design. No suppression needed. The estimator arms the tool by clicking; if they only want cycle navigation they will need to navigate to the target page first (acceptable trade-off per the UI spec).

### Pitfall 6: Wall arming skips height prompt — but popup still needs initialWallHeight
**What goes wrong:** When a wall item is armed and the user draws their first wall segment, the commit popup (`MarkupNamePopup` in `mode='save-after'`) appears. The `chainArmed: true` effect auto-commits WITHOUT showing the popup (CanvasViewport lines 518–539). So the user never sees the height field — they inherit the last used height.
**Why it happens:** The chain auto-commit bypasses the popup when `chainArmed && pendingName` is set. This is the CORRECT behavior per UI spec (7.1-I-06): wall height is inherited, no re-prompt.
**How to avoid:** No action needed. The `pendingWallHeight` set by `activatePreset(wall, { wallHeight })` is preserved through the chain auto-commit at line 352: `pendingWallHeight: payload.wallHeight ?? prev.pendingWallHeight`.
