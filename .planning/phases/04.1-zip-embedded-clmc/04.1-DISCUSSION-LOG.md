## Area 1: Revised PDF Workflow

**Context:** With PDF embedded in the ZIP, the Phase 04 "Browse for PDF" missing-PDF recovery modal becomes irrelevant. New scenario: architect sends a revised plan v2.pdf mid-bid.

### Q: How should the estimator update their project when receiving a revised PDF?

Options presented:
1. Replace Plan (in-app) — "Replace Plan PDF" toolbar button with validation
2. Start fresh each revision — no built-in replace
3. Defer to a later phase

**User selected:** Replace Plan (in-app)

**Notes:** User agreed losing markups on every revision is unacceptable mid-bid.

---

### Q: What checks should block or warn when replacing the plan PDF?

Options presented:
1. Page count must match; dimensions warn (recommended — mirrors Phase 04 D-26/D-27)
2. Page count must match; dimensions block too
3. No checks — just swap

**User selected:** Page count must match, dimensions warn (Recommended)

**Notes:** Keeps parity with Phase 04's proven validation logic.

---

### Q: Where should "Replace Plan PDF" live in the UI?

Options presented:
1. Toolbar button, next to Open/Save (recommended)
2. File menu only
3. Right-click on title bar

**User selected:** Toolbar button, next to Open/Save (Recommended)

## Area 2: Save UX for Large PDFs

**Context:** jszip STORE mode writes ~50MB/s. A 100MB PDF takes ~2s on a slow drive. First Save As is the worst case.

### Q: Should saving give any visual feedback while the ZIP is being written?

Options presented:
1. Brief "Saving..." in the title bar (recommended — reuses existing TitleBar wiring)
2. Reuse ConfirmationToast only (no in-flight indicator)
3. Disabled Save button only

**User selected:** Brief "Saving..." in the title bar (Recommended)

---

### Q: Should the Save button stay disabled while saving, as a guard against double-save?

Options presented:
1. Yes — disable Save + SaveAs during write (recommended)
2. No — title bar indicator is enough

**User selected:** Yes — disable Save + SaveAs during write (Recommended)

---

## Area 3: PDF Metadata in the Archive

**Context:** `absolutePath`/`relativePath` are removed. What metadata about the original PDF is retained?

### Q: What should the title bar and project display show as the "plan name"?

Options presented:
1. Original PDF filename, stored in project.json (recommended)
2. The .clmc filename only
3. No stored name — derive from .clmc path

**User selected:** Original PDF filename, stored in project.json (Recommended)

**Notes:** e.g. "level-1-plans.pdf" stored as `pdf.originalFilename`; used for display only, not file resolution.

---

### Q: Should we keep SHA256 of the embedded PDF in project.json?

Options presented:
1. Yes — store SHA256, check on open (recommended — corruption detection, reuses sha256File())
2. No — skip hash for embedded PDF

**User selected:** Yes — store SHA256, check on open (Recommended)

---

### Q: What should the PDF be named inside the ZIP archive?

Options presented:
1. Always "plan.pdf" (recommended — predictable, no filename edge cases)
2. Original filename preserved

**User selected:** Always "plan.pdf" (Recommended)

## Claude's Discretion Items

The following were not discussed — Claude handles them at planning/implementation time:

- **Migration detection mechanism:** Check for ZIP magic bytes `PK\x03\x04`. Plain-JSON v1 files auto-upgrade silently (mark dirty, no prompt).
- **jszip API details:** `generateAsync({ type: 'nodebuffer', compression: 'STORE' })` in main process.
- **IPC handler naming:** Extend or replace existing `file:readProject`/`file:writeProject`.
- **project.json formatting inside ZIP:** Recommend minified.
- **Save As default filename source:** Derive from `pdf.originalFilename` since `absolutePath` is gone.
- **`sha256Buffer()` helper:** May need a Buffer-based variant of `sha256File()`.
