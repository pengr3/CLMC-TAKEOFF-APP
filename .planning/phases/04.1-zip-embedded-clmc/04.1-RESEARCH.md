**Researched:** 2026-04-30
**Domain:** ZIP archive packaging in Electron main process (jszip) + Buffer-mode SHA256 + Buffer-over-IPC + magic-byte detection + silent v1‚Üív2 migration
**Confidence:** HIGH

---

## Summary

Phase 4.1 is a **localized format swap**, not a new-feature phase. The change touches three concerns: (1) replace plain-JSON write/read in `project-io.ts` with jszip-based ZIP assembly/extraction; (2) swap `sha256File(path)` for a `sha256Buffer(buf)` variant since the PDF bytes now live in memory at save time, not on disk; (3) restructure the IPC contract so `file:writeProject` accepts a `Uint8Array` PDF payload from the renderer, and `file:readProject` returns both the parsed `project.json` AND the embedded PDF bytes. Everything else ‚Äî the modal recovery flow, the dirty-flag plumbing, the keyboard shortcuts, the close guard, the snapshot/hydrate pure functions ‚Äî already exists and stays unchanged.

The new dependency is `jszip@3.10.1` (pure JS, ships its own TypeScript types ‚Äî no `@types/jszip` needed). It supports STORE compression (CONTEXT D-03), accepts Node Buffers directly via `.file(name, buf)`, and produces a `nodebuffer` output ready for `fs.writeFile`. There are no native addons, no electron-builder rebuilds, no platform-specific concerns. The only architectural call left to the planner is whether to evolve `file:readProject` / `file:writeProject` (recommended) or version them (`file:readProjectV2`, etc.). Given that v1 plain-JSON is a Phase 04 testing artifact only ‚Äî never shipped ‚Äî evolving in place is cleaner.

The single non-trivial design decision is **memory profile**. STORE mode + jszip means the entire ZIP is assembled in a Node Buffer in main, then written. For a 50 MB PDF that is fine; for a 200 MB scanned PDF this allocates ~200 MB transient. STATE.md flags PDFs up to ~200 MB as a known consideration. This matches existing `readPdfBytes` IPC behaviour (reads the whole PDF into an `ArrayBuffer`), so memory ceiling is unchanged. The path forward if it becomes a problem is `archiver` (stream-based), but that's a v2 concern ‚Äî flag and move on.

**Primary recommendation:** Install `jszip@3.10.1`. Add `sha256Buffer()` next to existing `sha256File()` in `project-io.ts`. Add `assembleClmcZip(projectJson: string, pdfBytes: Buffer)` and `extractClmcZip(zipBytes: Buffer)` helpers in the same file. Keep IPC handler names (`file:readProject` / `file:writeProject`) but evolve their signatures: write takes a `Uint8Array` PDF payload alongside the JSON; read returns `{ projectJson: string, pdfBytes: ArrayBuffer, isV1Legacy: boolean }`. Detect format via the first 4 bytes of the file (`PK\x03\x04` ‚Üí ZIP, anything else ‚Üí try JSON parse, fail ‚Üí corrupt). Pass PDF bytes across IPC as `Uint8Array` (Electron's structured-clone-based IPC handles this cleanly; Node Buffers auto-convert to `Uint8Array` on the renderer side, so be explicit about the wire type to avoid type confusion).

## 1. jszip API

### Library Choice and Version

| Property | Value |
|----------|-------|
| Package | `jszip` |
| Version (verified) | `3.10.1` (published 2022-08-02; current stable as of 2026-04-30 per `npm view jszip version`) |
| TypeScript types | Bundled (`types: './index.d.ts'` in package.json) ‚Äî **do NOT install `@types/jszip`** |
| Runtime dependencies | `lie@~3.3.0`, `pako@~1.0.2`, `setimmediate@^1.0.5`, `readable-stream@~2.3.6` (all pure JS, no native addons) |
| v4? | No v4 exists. v3.10.1 is the current stable. |
| Electron 35 / Node 22 compat | HIGH confidence ‚Äî pure JS, no platform-specific code paths used in our flow |

### Required API Surface

Three operations are needed: assemble a ZIP from `project.json` + PDF bytes, extract project.json + plan.pdf from a loaded ZIP, and detect non-ZIP input. All three are first-class jszip APIs.

#### 1.1 Assemble (save path, main process)

```typescript
// src/main/project-io.ts
import JSZip from 'jszip'

export async function assembleClmcZip(
  projectJson: string,      // already serialized via JSON.stringify(snap)
  pdfBytes: Buffer          // embedded PDF, name 'plan.pdf' per D-04
): Promise<Buffer> {
  const zip = new JSZip()
  // STORE compression per file (D-03) ‚Äî explicit per-file beats global because
  // it locks in the contract even if a future refactor adds a third entry.
  zip.file('project.json', projectJson, { compression: 'STORE' })
  zip.file('plan.pdf',    pdfBytes,    { compression: 'STORE' })
  return await zip.generateAsync({
    type: 'nodebuffer',
    compression: 'STORE'   // global default; matches per-file
  })
}
```

**Source:** [jszip generateAsync docs](https://stuk.github.io/jszip/documentation/api_jszip/generate_async.html), [jszip file() docs](https://stuk.github.io/jszip/documentation/api_jszip/file_data.html). HIGH confidence.

#### 1.2 Extract (open path, main process)

```typescript
// src/main/project-io.ts
export interface ExtractedClmc {
  projectJson: string       // raw text ‚Äî caller calls JSON.parse + validateV2
  pdfBytes: Buffer          // for IPC return + sha256Buffer compare
}

export async function extractClmcZip(zipBytes: Buffer): Promise<ExtractedClmc> {
  const zip = await JSZip.loadAsync(zipBytes)
  const jsonEntry = zip.file('project.json')
  const pdfEntry  = zip.file('plan.pdf')
  if (!jsonEntry) throw new Error('Invalid .clmc archive: missing project.json')
  if (!pdfEntry)  throw new Error('Invalid .clmc archive: missing plan.pdf')
  const [projectJson, pdfBytes] = await Promise.all([
    jsonEntry.async('string'),
    pdfEntry.async('nodebuffer')
  ])
  return { projectJson, pdfBytes }
}
```

**Source:** [jszip loadAsync docs](https://stuk.github.io/jszip/documentation/api_jszip/load_async.html), [jszip ZipObject.async docs](https://stuk.github.io/jszip/documentation/api_zipobject/async.html). HIGH confidence.

#### 1.3 Type Notes

`generateAsync` accepts these `type` values: `base64`, `binarystring`, `string`, `array`, `uint8array`, `arraybuffer`, `blob`, `nodebuffer`. Use `nodebuffer` in main (Node Buffer is what `fs.writeFile` expects).

`loadAsync` accepts: `String/Array of bytes/ArrayBuffer/Uint8Array/Buffer/Blob/Promise`. We pass a Node Buffer from `fs.readFile`.

`ZipObject.async(type)` accepts: `string` (or `text`), `nodebuffer`, `uint8array`, `arraybuffer`, `base64`, `binarystring`, `array`, `blob`. Use `string` for `project.json` and `nodebuffer` for `plan.pdf`.

Compression is set EITHER globally on `generateAsync` OR per-file on `.file()`. Per-file overrides global. Setting both (as in 1.1) is belt-and-braces and is what we want for a stable, audit-friendly format guarantee.

### Async vs Sync

jszip 3.x is **async-only** for generate/load (`generateAsync` / `loadAsync` return `Promise`). There is no sync API. This is fine for Electron's main process ‚Äî IPC handlers are already async ‚Äî and aligns with our existing pattern. Do NOT block the main process by awaiting in a tight loop.

### Verdict

jszip 3.10.1 covers every requirement of D-01 through D-07 with a clean three-line API surface. No alternative library beats it for this use case (alternatives evaluated: `archiver` is stream-based but write-only; `adm-zip` is sync and last published 2024 but smaller community; `yauzl`/`yazl` are read/write but lower-level). jszip wins on simplicity + type bundling + already-deployed-everywhere reputation.

## 2. SHA256 Buffer Hashing

The current `sha256File(path)` in `project-io.ts` lines 10‚Äì18 streams from disk. With v2, the PDF lives in memory at save time (renderer just embedded fresh bytes) and at load time (jszip just extracted bytes). We need a Buffer-input variant.

### Implementation

```typescript
// src/main/project-io.ts ‚Äî add alongside existing sha256File
import { createHash } from 'crypto'

/**
 * SHA256 of an in-memory Buffer. Returns a 64-char lowercase hex string.
 * Use this when the bytes already exist in memory (post-extract, post-embed).
 * For files on disk, prefer sha256File which streams.
 */
export function sha256Buffer(buf: Buffer | Uint8Array): string {
  return createHash('sha256').update(buf).digest('hex')
}
```

**Why sync is fine here:** `crypto.createHash().update().digest()` is sync but OpenSSL-backed. For a 50 MB Buffer, this is ~50‚Äì150 ms on a modern CPU ‚Äî single chunk, no event-loop yield needed. For a 200 MB Buffer, 200‚Äì600 ms. Worst-case still under 1 s. CONTEXT.md does not require streaming this; the bytes are already loaded by the time we hash.

**Why not stream?** We already have the full Buffer (jszip extracted it / renderer just sent it). Re-streaming would mean writing to a temp file then reading back, which is slower and gross.

**Performance reference (verified in `04-RESEARCH.md` Pitfall 3):** Node SHA256 throughput is 300‚Äì800 MB/s on modern SSDs. Our hash is now CPU-bound (no disk I/O), so the upper bound applies ‚Äî typical 50 MB PDF hashes in ~60‚Äì170 ms.

### Two Use Sites in Phase 4.1

1. **Save path:** Renderer holds `pdfBytes` (it loaded the PDF earlier or just received it from the Replace Plan PDF picker). Renderer sends bytes to main; main calls `sha256Buffer(pdfBytes)` and writes the result into `project.json` BEFORE assembling the ZIP. (Alternative: hash in renderer via WebCrypto. Rejected ‚Äî see ¬ß6 ‚Äî keeps crypto on one side.)

2. **Open path:** main extracts `plan.pdf` from the ZIP, calls `sha256Buffer(pdfBytes)`, compares against `project.json`'s `pdf.sha256`. Mismatch returns a `kind: 'archive-corrupted'` result (analogous to existing `kind: 'hash-mismatch'`).

### Test (Wave 0)

`src/tests/project-io.test.ts` already tests `sha256File`. Add:

```typescript

  const fileHash = await sha256File(FIXTURE)

```

## 3. ZIP Format Detection

CONTEXT D-06 requires silent v1 (plain JSON) ‚Üí v2 (ZIP) detection on open. The standard signature for a ZIP archive's first local-file-header record is `PK\x03\x04` ‚Äî bytes `0x50 0x4B 0x03 0x04`.

### Implementation

```typescript
// src/main/project-io.ts
const ZIP_MAGIC = Buffer.from([0x50, 0x4b, 0x03, 0x04]) // 'P','K',\x03,\x04

export type ClmcFormat = 'v2-zip' | 'v1-json' | 'unknown'

/**
 * Detect file format by sniffing the first 4 bytes.
 * 'PK\x03\x04' ‚Üí ZIP archive (v2)
 * Anything else that JSON-parses with formatVersion: 1 ‚Üí v1 plain JSON
 * Anything else ‚Üí unknown / corrupt
 *
 * Caller is responsible for the v1 JSON-parse step; this only checks the
 * magic bytes. Returns 'v1-json' for any non-ZIP buffer (caller verifies it parses).
 */
export function detectClmcFormat(buf: Buffer): ClmcFormat {
  if (buf.length < 4) return 'unknown'           // empty / truncated
  if (buf.subarray(0, 4).equals(ZIP_MAGIC)) return 'v2-zip'
  return 'v1-json'                                // tentative ‚Äî caller validates JSON
}
```

**Why `subarray` not `slice`:** `Buffer.subarray` returns a view (no allocation); `Buffer.slice` is deprecated in Node 22.

**Why `.equals` not byte-compare:** built-in `Buffer.equals` is one-line, compiles to native, returns boolean.

### Edge Cases

| Input | Detection | Caller behaviour |
|-------|-----------|------------------|
| Empty file (0 bytes) | `'unknown'` | Throw "Empty .clmc file" |
| 1‚Äì3 bytes | `'unknown'` | Throw "Truncated .clmc file" |
| Starts with `{` (plain JSON) | `'v1-json'` | Try JSON.parse + validateV1; fail ‚Üí throw "Corrupt project file" |
| Starts with `PK\x03\x04` (ZIP) | `'v2-zip'` | Pass to extractClmcZip; if extract throws, surface as corrupt-archive |
| Starts with `PK\x05\x06` (empty ZIP archive ‚Äî central directory only) | `'v1-json'` | Will fail JSON.parse ‚Üí "Corrupt project file" ‚Äî acceptable degradation |
| Random binary garbage | `'v1-json'` | Will fail JSON.parse ‚Üí "Corrupt project file" |
| UTF-8 BOM (`EF BB BF`) at start of v1 JSON | `'v1-json'` | Will fail JSON.parse on standard parsers; not a real concern (we wrote the file ourselves with no BOM) |

**The empty-ZIP case:** `PK\x05\x06` is the End-of-central-directory record signature. A genuinely empty ZIP archive uses ONLY this record (no local-file-header). We never produce empty ZIPs (we always write `project.json` + `plan.pdf`), so this only happens via external tampering. Our detector classifies it as `'v1-json'`, JSON-parse fails, user sees "Corrupt project file." Acceptable ‚Äî the file IS corrupt for our purposes.

**Confidence:** HIGH. The PK\x03\x04 signature is documented in the [PKWARE APPNOTE.txt](https://pkwarelabs.com/documents/casestudies/APPNOTE.TXT) ¬ß4.3.7 and is the universal first-bytes-of-a-non-empty-ZIP magic.

## 4. Electron IPC Buffer Transfer

### Key Findings (HIGH confidence)

Electron's `ipcRenderer.invoke` / `ipcMain.handle` use the [HTML Structured Clone Algorithm](https://html.spec.whatwg.org/multipage/structured-data.html#structured-clone). Per Electron's breaking-changes documentation:

- **ArrayBuffer**: transferred natively, efficiently (no serialization overhead beyond the structured-clone wrap)
- **Uint8Array, Int8Array, Float32Array, etc.** (TypedArrays): transferred as the same TypedArray on the other side
- **Node.js Buffer**: when sent renderer‚Üímain or main‚Üírenderer, **arrives as `Uint8Array` on the receiving side**. This is the canonical Electron behaviour. Buffer is a Node-only subclass of Uint8Array; structured-clone strips the Buffer prototype.
- **Functions, Promises, Symbols, WeakMap/Set**: throw on invoke
- **DOM types (File, Blob, ImageBitmap)**: depends on type ‚Äî Blob works in modern Electron but isn't relevant here

### Practical Wire Type for Phase 4.1

We'll send PDF bytes both directions across IPC. Use `Uint8Array` as the wire type ‚Äî explicit, symmetric, and survives structured clone unchanged:

| Direction | Sender holds | Wire type | Receiver receives |
|-----------|--------------|-----------|-------------------|
| save: renderer ‚Üí main | `Uint8Array` (already what `readPdfBytes` returns wrapped, or `new Uint8Array(arrayBuffer)`) | `Uint8Array` | `Uint8Array`. Convert to Node Buffer via `Buffer.from(u8.buffer, u8.byteOffset, u8.byteLength)` ‚Äî zero-copy view. |
| open: main ‚Üí renderer | Node Buffer from jszip's `async('nodebuffer')` | `Uint8Array` (re-wrap with `new Uint8Array(buf.buffer, buf.byteOffset, buf.byteLength)`, OR send the Buffer directly and let structured clone convert it) | `Uint8Array`. Then `pdfjsLib.getDocument({ data })` accepts both `Uint8Array` and `ArrayBuffer`. |

**Recommendation: always wire `Uint8Array`, both directions.** Two reasons: (a) the receive side gets the same shape regardless of what main sent, removing a class of "Buffer/Uint8Array confusion" bugs; (b) the existing `readPdfBytes` IPC handler already returns `ArrayBuffer` ‚Äî we keep the new handler API analogous and predictable.

The conversion in main is a zero-copy view:
```typescript
// renderer‚Üímain: Uint8Array arrives, wrap as Buffer for jszip
const pdfBuffer = Buffer.from(u8.buffer, u8.byteOffset, u8.byteLength)
```

**Source:** [Electron #9509 ‚Äî ArrayBuffer over IPC](https://github.com/electron/electron/issues/9509), [Electron #35152 ‚Äî Uint8Array ‚Üí Object bug](https://github.com/electron/electron/issues/35152), [Electron breaking-changes docs](https://www.electronjs.org/docs/latest/breaking-changes), [Electron ipcRenderer docs](https://www.electronjs.org/docs/latest/api/ipc-renderer). HIGH confidence.

### Size Limits

There is no documented hard size limit on Electron IPC payloads. Practical observations:

- **Memory cost:** Structured clone duplicates the bytes during the cross-process transfer. Sending a 50 MB Uint8Array means peak ~100 MB transient (sender holds + clone in transit). For 200 MB PDFs this peaks at ~400 MB transient ‚Äî within Electron's renderer memory ceiling but worth noting.
- **Wall-time cost:** ~50‚Äì100 ms per 50 MB transferred (anecdotal; varies by machine). Not user-perceptible for typical 10‚Äì50 MB PDFs.
- **Existing pattern:** `readPdfBytes` already does this (returns `ArrayBuffer`). We have empirical proof it works for the PDFs this app handles.

**Verdict:** No size-limit fallback needed for v1 of the format. If users hit memory issues with truly enormous (>500 MB) scans, the path forward is to write the PDF to a temp file in main, send only the path to renderer, and rewrite `loadPdfFromPath` to accept the temp path. Defer until reported.

### Per CONTEXT D-12: `isSaving` Flag Order

The async IPC round-trip during save is exactly where `isSaving = true` matters. Sequence:

```typescript
// renderer side, saveProject action
useProjectStore.setState({ isSaving: true })  // BEFORE the IPC call
try {
  await window.api.writeProjectV2(clmcPath, projectJsonStr, pdfBytesU8)
  useProjectStore.getState().setSaved(clmcPath)  // sets isDirty=false, isSaving handled below
} finally {
  useProjectStore.setState({ isSaving: false })  // ALWAYS resets, even on error
}
```

The TitleBar reads `isSaving`, shows "Saving..." while true. The Toolbar reads `isSaving`, disables Save / SaveAs / Replace Plan PDF buttons while true.

## 6. Memory Considerations

### Profile

The save-path peak memory is the worst case. Tracing it for a 50 MB PDF:

| Step | Memory held |
|------|-------------|
| 1. Renderer reads PDF bytes (initial open earlier) | 50 MB ArrayBuffer in renderer |
| 2. Renderer wraps in `Uint8Array` for IPC | 50 MB (same backing buffer, just a view) |
| 3. Renderer ‚Üí main IPC structured clone | +50 MB transient during clone |
| 4. Main: `Buffer.from(u8.buffer, ...)` (zero-copy view) | 50 MB |
| 5. Main: `JSZip.file('plan.pdf', buf)` | jszip stores a reference; no copy |
| 6. Main: `generateAsync({type:'nodebuffer', compression:'STORE'})` | Allocates the output ZIP Buffer ‚âà PDF size + ~few KB headers ‚âà 50 MB |
| 7. Main: `fs.writeFile(path, zipBuf)` | 50 MB held until write completes |

**Peak transient:** ~150 MB for a 50 MB PDF (renderer source + IPC clone + jszip output). For a 200 MB PDF: ~600 MB transient. Within Electron's process limits (typical default is 2‚Äì4 GB per process on 64-bit).

### Open path

| Step | Memory held |
|------|-------------|
| 1. Main: `fs.readFile(clmcPath)` ‚Üí Buffer | ~50 MB (ZIP file size) |
| 2. Main: `JSZip.loadAsync(buf)` parses the central directory | +small overhead |
| 3. Main: `pdfEntry.async('nodebuffer')` extracts PDF | +50 MB (extracted PDF Buffer) |
| 4. Main ‚Üí renderer IPC structured clone of pdfBytes | +50 MB transient |
| 5. Renderer: PDF.js `getDocument({data})` | +50 MB initially, then PDF.js builds its own structures |

**Peak transient:** ~150‚Äì200 MB for a 50 MB PDF. Same order of magnitude as the existing readPdfBytes flow.

### STORE Mode Memory

Per CONTEXT D-03, we use STORE (no compression). This means:
- jszip does NOT call pako/DEFLATE during generate ‚Äî saves CPU but not memory
- The output Buffer is always ‚â• input size (with STORE there's no compression)
- No streaming option in jszip 3.x for `generateAsync` ‚Äî the whole ZIP exists in memory before write

**Verdict:** Memory profile matches the existing readPdfBytes pattern. We're not introducing a new memory ceiling; we're keeping the same one. If the user hits OOM with a 500 MB PDF, both the existing flow (open) AND the new flow (save) fail ‚Äî but the user already has bigger problems (PDF.js itself struggles with 500 MB scans).

### Future Path (out of scope for v1 of this format)

If memory becomes a problem:
- **`archiver` library**: stream-based ZIP write, never holds full archive in memory
- **Temp file streaming**: write PDF to a temp file in main, stream-add to archive, delete temp
- **STORE-mode optimization**: STORE writes the file bytes verbatim ‚Äî could in principle be done with manual ZIP writing (a 30-byte local-file-header + PDF bytes + central directory) without jszip at all

None of these are needed for v1 of the format. Flag in ¬ß10 Open Questions.

### Key Planning Considerations

1. **Naming choice for evolved IPC channels:** Recommend keeping `file:readProject` / `file:writeProject` (evolve in place) over `file:readProjectV2` / `file:writeProjectV2` (versioned). Reasons: v1 plain-JSON was a Phase 04 testing artifact only, no shipping users; the discriminated-union return type from `readProject` cleanly handles BOTH formats; versioning would proliferate channels for no gain.

2. **Where to hash the embedded PDF ‚Äî main or renderer?** Recommend hash in main (after extract on open; via direct call to `sha256Buffer` on the PDF bytes the renderer sends on save). Keeps crypto concerns on one side. Optional: also expose a `file:sha256Buffer` IPC channel if future use cases need it (currently no ‚Äî the renderer never needs to hash).

3. **`hashPdf` IPC channel ‚Äî keep or delete?** Recommend delete after Wave 4. Phase 04 used it three places: hash on save (now in-main via `sha256Buffer`), hash on open (now in-main on extracted bytes), hash on relink (no more relink). All three replaced. Delete during Wave 2 when evolving IPC handlers.

4. **v1 `MissingPdfModal` ‚Üí does the v1 silent-upgrade path need it?** Question: when opening a v1 file whose absolutePath doesn't resolve, do we still show MissingPdfModal? **Recommendation: NO ‚Äî and prefer aggressive simplification.** v1 files are dev-only Phase 04 artifacts. Document that "Phase 04 .clmc files require their original PDF to still be at the saved absolute path; otherwise re-create the project from the PDF." Delete `MissingPdfModal` entirely. If user reports they have a v1 file with a missing PDF, the answer is "open the PDF first, then redo the markup." This is brutal but right ‚Äî keeps Phase 4.1 cleanly forward-looking.

5. **Pretty-print or minified `project.json` inside the ZIP?** CONTEXT recommends minified. Recommend follow CONTEXT ‚Äî the size difference is trivial (project.json with full markups for a 1000-pin job is ~100 KB; pretty-print adds ~10‚Äì20% which is invisible next to a 50 MB PDF). Counter-argument: pretty-print enables ZIP-renamed-to-zip inspection. Verdict: minified for production write; if dev-debug becomes important, add a `__DEV__` flag later. Keep `JSON.stringify(snap)` (no second arg) per CONTEXT.

6. **`isSaving` flag ‚Äî two-action API or single setter?** Recommend single `setSaving(bool)` action. Mirrors `markDirty()` pattern. Caller wraps the IPC call in try/finally to guarantee reset on error.

7. **Atomic write?** ZIP write to disk is NOT atomic ‚Äî a crash mid-write produces a truncated ZIP that detectClmcFormat would still classify as `'v2-zip'` (PK\x03\x04 prefix), but jszip.loadAsync would throw. Defensive option: write to `${clmcPath}.tmp`, rename to `${clmcPath}` on success. Recommend: defer atomic writes to a future polish phase. Native `fs.writeFile` is fine for v1 ‚Äî Node's writeFile uses a single write syscall on Windows, so partial writes only happen on power loss. Manual checkpoint after each save in QA covers the workflow.

8. **Replace Plan PDF ‚Äî does it need a confirm modal before page-count + dim checks?** No ‚Äî page-count abort and dim warn ARE the confirm gates. If both pass cleanly, just embed and mark dirty. The UX is: click Replace ‚Üí file picker ‚Üí instant swap (if compatible) or modal (if not).

9. **PDF bytes caching in renderer:** Currently `viewerStore.pdfDocument` holds a `PDFDocumentProxy` (parsed PDF) but NOT the raw bytes. To save without re-reading from disk, the bytes need to be retained. Recommend: add `pdfBytes: Uint8Array | null` to viewerStore alongside `pdfDocument`. Set both atomically in `usePdfDocument.loadPdf`. On save, read `pdfBytes` from viewerStore. (Memory cost: same bytes already held by pdfjs anyway; this is a reference, not a copy.)

10. **Backward-compat for existing test fixtures:** `src/tests/fixtures/sample-1page.pdf` exists and is used by `project-io.test.ts` for sha256 testing. Reuse it for assemble/extract tests too (acts as the embedded PDF in synthetic .clmc files). No new fixtures needed.

### Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|-----------|
| jszip 3.10.1 has a bug with large Buffers in Node 22 | HIGH (save fails for big PDFs) | Wave 0 includes a test that round-trips the existing 50+ MB-equivalent fixture; if it fails, escalate before further work |
| Buffer/Uint8Array confusion across IPC | MEDIUM (subtle bytes-mangled bugs) | Wire `Uint8Array` always; explicit `Buffer.from(u8.buffer, u8.byteOffset, u8.byteLength)` zero-copy view in main |
| PDF.js can't render bytes loaded via `getDocument({data: Uint8Array})` after IPC clone | LOW (existing `readPdfBytes` proves it works) | Wave 4 integration test: open a v2 file end-to-end |
| User has a Phase 04 .clmc file with absolute path that no longer exists, AND we deleted MissingPdfModal | MEDIUM (existing file becomes unopenable) | Document the limitation in release notes; fallback: keep a thin v1-PDF-not-found error message in OpenErrorModal |
| Memory peak ~600 MB for a 200 MB PDF | LOW for typical use (10‚Äì50 MB), MEDIUM for edge-case scans | Document; defer `archiver`/streaming until reported |
| ZIP file partial-write on power loss leaves corrupt .clmc | LOW | Defer atomic rename to polish phase |

### Open Questions (for planner to resolve)

1. **v1 silent-upgrade scope:** Will v1 plain-JSON .clmc files exist in the wild beyond Phase 04 dev testing? If yes, we need to keep the v1 missing-PDF recovery flow. **Recommendation: assume NO. v1 was internal testing only. Simplify ruthlessly.**

2. **`pdfBytes` in viewerStore:** Where exactly should the raw bytes be cached? Options: (a) viewerStore alongside pdfDocument; (b) separate "pdfBytesStore"; (c) re-read from disk on save. **Recommendation: (a) ‚Äî same store, keeps the PDF artifact unified.** Cost: 50 MB held in memory while project is open. Acceptable.

3. **Confirmation toast wording:** "Saved" works for save-in-place. "Saved as plans.clmc" for save-as. Use the existing pattern from Phase 04 (App.tsx already does this). No change.

4. **Icon for Replace Plan PDF button:** Choose from `lucide-react`. Candidates: `RefreshCw`, `FileSearch`, `FilePlus`, `Replace`. **Recommend `Replace` if exported by lucide-react@1.6.0 in this version, otherwise `RefreshCw`.** Verify in planning.

5. **Should `ArchiveCorruptedModal` keep [Open anyway] or only [Cancel]?** D-07 says "Open anyway" exists. But "open anyway" on a sha mismatch where the PDF was extracted from inside the archive is suspicious ‚Äî there's no plausible "user just wants to ignore this" scenario unless we're worried about hash determinism bugs in our own code. **Recommend keep [Open anyway] per D-07** ‚Äî defensive escape hatch for any sha bug we haven't hit yet. Bug would surface clearly because the modal would appear on a freshly-saved file.
