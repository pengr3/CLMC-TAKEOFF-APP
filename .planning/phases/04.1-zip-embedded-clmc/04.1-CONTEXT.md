# Phase 4.1: ZIP-Embedded .clmc Format - Context

**Gathered:** 2026-04-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the plain UTF-8 JSON `.clmc` format (Phase 04) with a ZIP archive that bundles `project.json` + the embedded PDF as `plan.pdf`. The result is a fully self-contained file that opens on any machine without path resolution or missing-PDF recovery flows.

**In scope:** Format change (v1 plain JSON → v2 ZIP), silent migration on open for old v1 files, "Replace Plan PDF" toolbar action for architect revisions, save-in-flight UX (title bar + button disable).

**Out of scope:** Recent-files list, welcome screen, auto-save/recovery, cloud sync, open-without-PDF read-only mode, multi-window editing, BOQ export (Phase 5), running totals (Phase 6).

</domain>

<decisions>
## Implementation Decisions

### File Format

- **D-01:** `.clmc` v2 format is a ZIP archive using **jszip** (pure JS, no native addon). The archive contains exactly two entries: `project.json` (project data) and `plan.pdf` (the embedded PDF bytes).
- **D-02:** `formatVersion` bumps from `1` to `2`. The `pdf` block in `project.json` loses `absolutePath`, `relativePath`, and gains `originalFilename: string` (for display only). `pdf.sha256` is retained and now hashes the embedded bytes.
- **D-03:** PDF is stored in the ZIP using **STORE mode (no compression)** — construction PDFs are already internally compressed. This minimises CPU overhead on save and load.
- **D-04:** The PDF is always named `plan.pdf` inside the ZIP archive, regardless of the original filename. The original PDF filename (e.g. `level-1-ground-floor.pdf`) is preserved as `pdf.originalFilename` in `project.json` for title bar display only — not used for file resolution.
- **D-05:** `pdf.absolutePath` and `pdf.relativePath` fields from v1 are **removed** from `project.json`. Path resolution helpers `resolvePdfPath()` and `computeRelativePath()` in `project-io.ts` can be deleted.

### Migration from v1 Files

- **D-06:** On open, detect ZIP vs plain JSON by checking for the ZIP magic bytes `PK\x03\x04` at the start of the file. Plain-JSON v1 files: parse using the v1 schema, load into stores as normal, then **mark dirty immediately**. No migration prompt — the upgrade is silent. User must Ctrl+S to write the new v2 ZIP format. This is Claude's discretion on exact detection mechanism.

### Integrity Check

- **D-07:** SHA256 of the embedded `plan.pdf` is computed when the PDF is first embedded and stored as `pdf.sha256` in `project.json`. On open, the embedded PDF bytes are re-hashed and compared. A mismatch triggers a non-blocking warning modal ("Archive may be corrupted — open anyway? [Open] [Cancel]"). Reuses `sha256File()` logic from `project-io.ts` (may need a `sha256Buffer()` variant operating on a Buffer rather than a file path).

### PDF Revision

- **D-08:** A **"Replace Plan PDF" button** is added to the Toolbar, next to the existing Open/Save cluster.
- **D-09:** Replacement validation mirrors Phase 04's proven checks: page count mismatch → **hard abort** with error modal ("Selected PDF has N pages, project expects M. [Pick again] [Cancel]"). Page dimension mismatch → **warn and allow** ("PDF dimensions don't match original — markup positions may look wrong. [Open anyway] [Cancel]"). This keeps parity with Phase 04's D-26/D-27.
- **D-10:** On successful replace: embed new PDF bytes as `plan.pdf`, update `pdf.originalFilename`, update `pdf.sha256`, mark dirty. No automatic save — user must Ctrl+S to persist.

### Save UX

- **D-11:** While the ZIP write is in flight (async), the **title bar shows "Saving..."** and the **Save + SaveAs buttons are disabled**. On success: title bar reverts to the filename, `ConfirmationToast` fires ("Saved" / "Saved as {filename}"). The "Replace Plan PDF" button follows the same in-flight disable pattern.
- **D-12:** `projectStore` gains an `isSaving: boolean` field that drives the title bar text and button disabled state. Set to `true` before the IPC write call; set to `false` (and call `setSaved()`) in the resolved callback.

### Claude's Discretion

- jszip API: `generateAsync({ type: 'nodebuffer', compression: 'STORE' })` in the main process. Write result via `fs.writeFile`. Never generate the ZIP in the renderer.
- IPC handler naming: extend or replace existing `file:readProject` / `file:writeProject` — follow existing naming convention. Claude decides whether to evolve the same handler names or introduce `file:readProjectV2` / `file:writeProjectV2`.
- Whether `project.json` inside the ZIP is minified or pretty-printed — recommend minified (trivial size difference; project.json is tiny next to the PDF).
- Save As default filename: unchanged behaviour — PDF basename + `.clmc`. Since `absolutePath` is gone, derive from `pdf.originalFilename` stored in `projectStore`/`viewerStore` at the time of the first save.
- Exact shape of the `pdf` block in v2 `project.json` — minimum fields: `{ originalFilename, sha256, totalPages }`. Claude may add `dimensions` per-page if useful for the D-09 validation.

</decisions>

- `ConfirmationToast.tsx` — reuse unchanged for "Saved" / "Saved as {filename}" toast.
- `enforceClmcExtension()` in `project-io.ts` — unchanged; still needed for Save As.
- `useKeyboardShortcuts.ts` `isTextInputActive()` guard — already guards Ctrl+S; no change needed.

### Established Patterns
- IPC triad: every new IPC surface requires changes to `ipc-handlers.ts` (main), `preload/index.ts` (bridge), and `preload/index.d.ts` (types). No exceptions.
- Zustand `isSaving` pattern: follow same primitive-field selector convention as `isDirty` — `useProjectStore(s => s.isSaving)`.
- Modals follow `ScalePopup` inline-style pattern using `COLORS` constants; no new modal library.
- Parent-owned lifecycle for toasts and popups — the component mounting the toast owns its dismiss timer.

### Integration Points
- `Toolbar.tsx` — "Replace Plan PDF" button goes in the left cluster (Open / Save / Save As / **Replace Plan PDF**). Disable during `isSaving`.
- `TitleBar.tsx` — currently derives title from `projectStore.currentFilePath` and `isDirty`. Add branch: if `isSaving` → show "Saving..." prefix instead of filename.
- `ipc-handlers.ts` — `file:readProject` reads ZIP bytes, extracts `project.json` text + `plan.pdf` buffer, returns both to renderer. `file:writeProject` accepts serialized project JSON + PDF buffer, assembles ZIP, writes to disk.
- `project-io.ts` — remove path-resolution functions; add `sha256Buffer()`; possibly add `detectFormat(buffer)` returning `'v1-json' | 'v2-zip'`.

<specifics>
## Specific Ideas

- "Single file any user can open on any machine" — the core motivation. No setup, no re-linking, no "PDF not found". Send the `.clmc` and the recipient opens it directly.
- "ZIP is inspectable" — a user can rename `.clmc` to `.zip` and peek inside. `project.json` is readable; `plan.pdf` is extractable. This is a feature, not a concern.
- "Replace Plan PDF" handles the architect-revision workflow — estimators frequently receive revised plan sets mid-bid. Losing markups on each revision is unacceptable; the replace flow preserves all markup work while swapping the underlying plan.
- "Silent migration" — plain-JSON v1 files from Phase 04 testing should just open and work. No dialog, no confusion. The user will notice the format upgraded because the file size grows (PDF embedded) when they next save.

</specifics>
