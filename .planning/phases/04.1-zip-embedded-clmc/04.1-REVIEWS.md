# Cross-AI Plan Review — Phase 4.1

## Gemini Review

This is a high-quality, professional implementation plan that effectively addresses the complexities of transitioning to a self-contained file format. The use of TDD (Plan 00) and a clear separation of concerns (Logic → IPC → State → Hooks → UI) demonstrates a mature engineering approach.

### Summary

The plan provides a robust strategy for upgrading the project format to a ZIP-embedded structure using `jszip`. It prioritizes data integrity through SHA256 hashing and ensures a smooth user experience via silent migration from v1 to v2. The "Replace Plan PDF" workflow is well-guarded with page-count and dimension validation. By removing absolute/relative path resolution, the plan significantly reduces the complexity and brittleness of the application's file handling.

### Strengths

- **TDD-First Approach:** Establishing RED tests (Plan 00) for edge cases like ZIP magic byte detection and hash mismatches before writing implementation code is excellent for stability.
- **Performance Optimization:** Choosing `STORE` mode (no compression) for `jszip` is the correct technical decision, as PDFs are already compressed; this minimises CPU overhead and save/load times.
- **Migration Path:** The "silent upgrade" (D-06) for v1 files (load + mark dirty) is the most user-friendly way to handle breaking format changes.
- **Validation Logic:** Hard-aborting on page count mismatch during PDF replacement prevents fatal state corruption where markups would refer to non-existent pages.
- **IPC Cleanup:** Deleting obsolete IPC channels for path resolution (Plan 02) reduces surface area and technical debt.

### Concerns

- **Memory Footprint (HIGH):** Construction PDFs can exceed 100MB. Storing `pdfBytes` in `viewerStore` (Zustand) means the entire file resides in the renderer's memory twice (once in the store, once in the PDF.js internal heap). On low-RAM Windows machines, this could lead to OOM (Out of Memory) crashes.
- **Atomic Write Safety (MEDIUM):** ZIP generation and disk I/O are non-atomic. If the app crashes or the system loses power during `writeProject`, the `.clmc` file could be corrupted/truncated.
- **Scale Calibration Drift (MEDIUM):** When replacing a PDF with "dimension mismatch = warn+allow" (D-09), the existing scale calibration (pixels-to-meters) will likely be invalid if the new PDF has a different DPI or crop, even if the drawing content is "the same."
- **IPC Message Limits (LOW):** While Chromium handles large IPC messages, very large PDFs (200MB+) might hit performance bottlenecks or memory fragmentation during the structured clone process between Main and Renderer.

### Suggestions

- **Scale Reset Warning:** In the `ReplacePlanPdfResult` workflow, if a dimension mismatch occurs, explicitly warn the user that their **Scale Calibration** may need to be re-verified or reset.
- **Atomic Saving:** In `main/project-io.ts`, implement a "write-then-rename" strategy. Write the ZIP to a temporary file (e.g., `project.clmc.tmp`) and only rename it to the final destination once the write is successful. This prevents partial/corrupted saves from destroying the user's existing work.
- **Memory Management:** Consider if `viewerStore.pdfBytes` is strictly necessary after the initial load. If the project isn't dirty, could you re-read the bytes from disk? (Note: Since the ZIP is the source of truth, you'd have to re-extract, which might be slower. The current approach is faster but memory-heavy).
- **JSZip Buffer Handling:** Ensure `jszip.generateAsync({type: 'nodebuffer'})` is used in the Main process and `Uint8Array` in the Renderer to avoid unnecessary conversions between Node `Buffer` and Browser `ArrayBuffer`.

### Risk Assessment

**Risk Level: LOW**

The plan is exceptionally thorough. The primary risks are environmental (memory constraints and file system reliability) rather than logic-based. The migration strategy and validation guards effectively mitigate the risk of data loss. By locking in interfaces via RED tests early, the implementation is unlikely to diverge from the requirements.

**Justification:** The transition from external path references to embedded assets is a significant reduction in "state of the world" complexity, making the app much more robust for the end-user. The trade-off of increased memory usage is acceptable for a desktop application of this nature, provided the Atomic Write safety is considered.

## Internal Review (Orchestrating Model — Independent Analysis)

### Summary

The 7-plan wave structure is well-conceived: each wave has a minimal, testable blast radius and clean depends_on relationships. The TDD setup in Wave 0 is thorough and the IPC discriminated union in Wave 2 is a solid wire contract. One significant design gap stands out: the v1 silent-migration open path in Plan 04 does not explicitly address how `viewerStore.pdfBytes` gets populated when opening a v1 file (PDF bytes live on disk at v1.pdf.absolutePath, not in the ZIP). If left unaddressed, Ctrl+S after opening a v1 file would attempt to assemble a ZIP with `null` bytes and either crash or produce a broken file.

### Strengths

- **Wave isolation is correct:** Logic (W1) → IPC (W2) → State (W3) → Hooks (W4) → UI (W5) means each wave has a clear integration point and won't require backtracking.
- **RED test suite locks contracts early:** 11 test additions across 9 files in Wave 0 ensures implementation drift is caught immediately. The interface shapes in `<interfaces>` blocks are authoritative.
- **STORE compression is the right call:** Agreed with Gemini — construction PDFs are already DEFLATE-compressed; adding STORE mode also means round-trip sizes are predictable (ZIP overhead ~30–50 bytes per entry, so output ≈ PDF size + tiny JSON), which makes file-size behaviour obvious to users.
- **detectClmcFormat is cheap and safe:** 4-byte magic sniff is correct and fast. The fallback-to-v1-json for anything that isn't ZIP is appropriately permissive.
- **Plan 03 dirty-tracking invariant is correctly stated:** pdfBytes must NOT trigger isDirty. The subscribeWithSelector selector already excludes pdfDocument; extending that same exclusion to pdfBytes is straightforward and noted.
- **Path traversal mitigation in Plan 01:** Hard-coding ZIP entry names (`zip.file('project.json')` exact lookup) is the right approach — never iterate-and-trust arbitrary entry names.

### Concerns

- **V1 migration + pdfBytes population gap (HIGH):** Plan 04 states the v1-json branch does `migrate + hydrateStores + markDirty`. But it does NOT explicitly state how `viewerStore.pdfBytes` is populated. The v1 data includes `pdf.absolutePath` pointing to the external PDF. After `validateV1 → migrateV1ToV2`, `absolutePath` is dropped. The hook MUST extract `absolutePath` from the pre-migration v1 data and call `readPdfBytes(absolutePath)` to fill `viewerStore.pdfBytes` BEFORE migration discards the path. If this step is skipped, the next `saveProject` call reads `viewerStore.getState().pdfBytes` as `null`, and `assembleClmcZip` receives `null` bytes, producing a corrupt or crashed save. **Plan 04's v1-json branch must be written to: (1) parse raw JSON → get v1.pdf.absolutePath, (2) call `window.api.readPdfBytes(absolutePath)` to load raw bytes, (3) call `viewerStore.setPdfBytes(bytes)`, then (4) migrate+hydrate+markDirty.** This ordering is critical and is absent from the plan's description. Recommend adding a task to Plan 04 that explicitly calls this out, and adding a RED test to Plan 00 for this scenario.
- **sha256-renderer SubtleCrypto is async and slow for large PDFs (MEDIUM):** The renderer-side hash check (D-07) uses SubtleCrypto.digest() via `sha256-renderer.ts`. SubtleCrypto is web-standard async and runs off the main thread, but for a 100MB+ PDF, this still takes 200–800ms and blocks the open UX visibly. Alternative: perform the SHA256 comparison in the main process where Node's synchronous `createHash` already hashes the buffer cheaply in the `extractClmcZip` flow. The IPC handler could return `computedSha256` alongside `pdfBytes` in the `v2-zip` result, and the renderer just compares string values rather than hashing again. This would eliminate sha256-renderer.ts entirely and reduce Wave 4 scope.
- **Atomic write safety not covered (MEDIUM):** Agreed with Gemini. `fs.writeFile` is not atomic — a crash mid-write leaves a corrupt file. A write-then-rename (`fs.writeFile(path + '.tmp') → fs.rename(tmp, path)`) approach in `assembleClmcZip` or the IPC handler would protect the existing file until the new one is complete. This is a single line change and low-effort insurance for an estimator's irreplaceable work.
- **Replace + Save As default filename when pdfBytes came from replacement (LOW):** After a Replace Plan PDF, the `pdf.originalFilename` is updated to the new file's basename. The Save As suggested filename should derive from `pdf.originalFilename` (project's displayed PDF name), not from `currentFilePath`. The plans mention this but don't trace the exact path through the code — verify that `enforceClmcExtension` + the Save As dialog default are reading `viewerStore.pdfBytes.filename` or `projectStore.pdf.originalFilename` correctly.
- **DimensionMismatchModal scale warning gap (LOW):** When the user proceeds past a dimension mismatch in Replace flow, scale calibration is invalidated but no warning is surfaced in the modal. The plan's D-09 says "warn and allow" — recommend making the DimensionMismatchModal body text include: "Note: your scale calibration may need to be re-set after opening." This is a one-sentence UI addition.
- **Plan 04 is the highest-complexity plan (LOW note):** Plan 04 touches 6 files, deletes 2 components, and wires the most complex async flow. It should be treated as the highest-regression-risk plan in the phase. The manual UAT in Plan 06 directly validates this, but consider whether Plan 04's RED tests (replace-plan-pdf.test.ts, project-open-flow.test.ts) cover the v1 migration path specifically.
- **IPC Uint8Array confusion note (LOW):** Plan 02 correctly identifies that `pdfBytes` must cross the IPC boundary as `Uint8Array`. The implementation note says "main converts via `Buffer.from(u8.buffer, u8.byteOffset, u8.byteLength)`". This is correct but only safe when the Uint8Array is a view of an ArrayBuffer without additional byte offset (i.e., created directly from a Buffer). If the renderer sends a sub-view Uint8Array, the byteOffset must be included. The plan mentions this; ensure the implementation follows it exactly.

### Suggestions

- **Plan 00:** Add one more RED test to `replace-plan-pdf.test.ts` specifically for the v1 migration case: after `openClmcFromPath` resolves a v1 file, `viewerStore.getState().pdfBytes` should be a non-null Uint8Array. This will fail RED until Plan 04 explicitly populates it.
- **Plan 01 (`project-io.ts`):** Add a `sha256Buffer` comparison helper alongside `assembleClmcZip` that returns `boolean` — something like `verifySha256(pdfBytes: Buffer, expectedHex: string): boolean`. This makes the corruption check in Plan 04 a single call rather than requiring the caller to string-compare manually.
- **Plan 02 (IPC):** Add `computedSha256: string` to the `v2-zip` result object so the renderer can compare without re-hashing. Eliminates `sha256-renderer.ts` and Wave 4's SubtleCrypto dependency entirely.
- **Plan 03 (state):** Add an explicit note (or a `// never null after first PDF load` comment) on `viewerStore.pdfBytes`. The field should be treated as required once a project is open — make the type `Uint8Array | null` but document the lifecycle: `null` only before any PDF is loaded or after `reset()`.
- **Plan 04:** Add a dedicated Task that covers the v1-json branch pdfBytes population step explicitly. Something like: "For v1-json: before calling migrate(), extract v1Data.pdf.absolutePath and call `window.api.readPdfBytes(absolutePath)`, then `viewerStore.setPdfBytes(bytes)`. Proceed to migrate+hydrate+markDirty."
- **Atomic save:** In `assembleClmcZip` or the IPC `file:writeProject` handler, write to `${path}.tmp` then `fs.rename` to `path`. This is a 2-line change in `ipc-handlers.ts` with significant resilience payoff.

### Risk Assessment

**Risk Level: LOW–MEDIUM**

The plan is well-structured and the wave decomposition is sound. The HIGH concern (v1 pdfBytes gap) is a design oversight in Plan 04's description, not a fundamental flaw — it's fixable before execution by clarifying the task. The MEDIUM concerns (atomic write, sha256 renderer complexity) are both low-effort fixes that would make the implementation more robust. Without addressing the HIGH concern, the v1 migration path would silently break saves after migration, which would be a regressive user experience.

## Consensus Summary

Both reviewers found the overall plan structure sound and the TDD-first approach a strength. No reviewer flagged scope creep — the phase is appropriately scoped to the format change only.

### Agreed Strengths

- TDD-first (Wave 0 RED tests) is the right approach; locks contracts before implementation
- STORE compression is the correct performance decision for pre-compressed PDFs
- Silent v1 migration (no dialog, just mark dirty) is the best UX choice
- Deleting obsolete IPC channels (checkExists, resolvePdfPath, computeRelativePath, hashPdf) reduces debt
- Hard-abort on page count mismatch during Replace is correctly stringent

### Agreed Concerns

1. **Memory pressure from pdfBytes in Zustand (MEDIUM–HIGH):** Large PDFs (100MB+) double-buffered in renderer memory. For v1 desktop app this is acceptable but should be noted in the UAT checklist.

2. **Atomic write safety (MEDIUM):** No write-then-rename protection. A crash mid-save destroys the existing file. Low-effort fix: write `.clmc.tmp` then `fs.rename`.

3. **Scale calibration drift on dimension-mismatch Replace (MEDIUM):** The DimensionMismatchModal should warn that scale calibration needs re-verification.

### Agreed Suggestions (in priority order)

1. **BEFORE Plan 04 execution:** Clarify that the v1-json open path must read PDF bytes from `v1.pdf.absolutePath` and populate `viewerStore.pdfBytes` BEFORE discarding absolutePath via migration. Add a RED test for this.

2. **In Plan 01 or 02:** Add write-then-rename atomic save protection in the IPC `writeProject` handler.

3. **In Plan 02:** Consider returning `computedSha256` in the `v2-zip` IPC result to eliminate the `sha256-renderer.ts` SubtleCrypto dependency in the renderer.

4. **In Plan 04's DimensionMismatchModal:** Add one sentence warning about scale calibration invalidation.

### Divergent Views

- Gemini assessed Risk Level as LOW. Internal review assessed LOW–MEDIUM due to the v1 pdfBytes population gap in Plan 04 (which Gemini did not flag). After clarifying Plan 04 to address this, overall risk would be LOW.
- Gemini suggested keeping `viewerStore.pdfBytes` and managing it carefully. Internal review suggested eliminating `sha256-renderer.ts` by moving the hash computation to the IPC handler. Both are valid approaches; the IPC hash approach is simpler.
