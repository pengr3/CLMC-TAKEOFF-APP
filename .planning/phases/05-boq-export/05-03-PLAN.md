<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer (window.api.*) → preload bridge → ipcMain.handle | Untrusted renderer-supplied filePath + BoqStructure cross here; main MUST validate filePath via dialog (already done) but trusts BoqStructure shape (TS contract enforced by triad) |
| BoqStructure → buildBoqXlsx / buildBoqCsv → atomicWriteFile | Within main process — no boundary; native call chain |
| atomicWriteFile → fs.writeFile + fs.rename | Trusted Node.js APIs; OS-level disk semantics |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | Tampering | Renderer crafts a filePath like `C:/Windows/System32/cmd.exe` and calls writeBoqXlsx — overwrites a system file | mitigate | filePath ALWAYS comes from `dialog.showSaveDialog` which is OS-validated (user-confirmed save target). Renderer cannot forge a path that bypassed the dialog because the dialog handler returns the path as a string and the renderer round-trips that exact string back to the write handler. If a malicious renderer literally hardcoded a system path and skipped the dialog, that's a renderer-process compromise — the IPC layer is not the right defense (in Electron, the renderer has full window.api access already). The dialog is the trust boundary, not the IPC channel. |
| T-05-03-02 | Tampering | Path traversal: filePath = `../../etc/passwd.xlsx` after enforceXlsxExtension | mitigate | Same as above — `dialog.showSaveDialog` returns absolute paths normalized by the OS; the user saw and confirmed the target directory in the native dialog. enforceXlsxExtension only manipulates the trailing extension, not directory components. |
| T-05-03-03 | Denial of Service | Crash on corrupt aggregator state (BoqStructure with circular refs, NaN, missing fields) | mitigate | All three handlers wrap the writer + atomicWriteFile in `try/catch` and return `{ ok: false, reason: err.message }`. Renderer surfaces in OpenErrorModal-style. Never throw across the IPC boundary — fail-soft per Phase 4.1 D-21 pattern (matches existing `file:readProject` shape in ipc-handlers.ts:106-133). |
| T-05-03-04 | Information Disclosure | Error message in `reason` exposes filesystem paths or stack traces | accept | OS errors typically include the target path (`EACCES: permission denied, open 'C:/Windows/System32/...'`) which is exactly what the user needs to debug. No secrets are added by us. |
| T-05-03-05 | Tampering | Atomic write `.tmp` orphan left on disk if rename fails | mitigate | Existing atomicWriteFile at ipc-handlers.ts:45-55 already cleans up on rename error via best-effort `unlink`. Wave 0 IPC test asserts `unlink` is called on rename failure. |
| T-05-03-06 | Tampering | filterIndex missing from showSaveDialog → wrong format detected → write XLSX bytes to a .csv file | mitigate | Per RESEARCH §5 + Pitfall 7: Electron's showSaveDialog does NOT return filterIndex. Detect format from `result.filePath`'s extension (`.toLowerCase().endsWith('.csv')`) and enforce the matching extension. Wave 0 test asserts both `.xlsx` and `.csv` paths are correctly classified and round-tripped. |
| T-05-03-07 | Spoofing | Renderer calls writeBoqXlsx with a CSV-format BoqStructure (e.g., grandTotals shaped wrong) | accept | TS contract via the inline duplicated types catches structural mismatches at compile time in both renderer and main. At runtime, ExcelJS handles malformed input gracefully (skips undefined cells); worst case is a partial workbook — atomicWriteFile is NOT atomic against logical correctness, only against partial-byte writes. Acceptable for v1. |
</threat_model>

```typescript
  // -------- Phase 5: BOQ Export — D-16 / D-24 --------
  // Save dialog with .xlsx and .csv filters; detects format from chosen extension
  // (Pitfall 7: showSaveDialog does NOT return filterIndex).
  ipcMain.handle(
    'dialog:saveExport',
    async (
      event,
      defaultPath: string,
      format: 'xlsx' | 'csv'
    ): Promise<{ filePath: string; format: 'xlsx' | 'csv' } | null> => {
      const win = BrowserWindow.fromWebContents(event.sender)!
      // Filter order respects user's preferred default — primary filter listed first.
      // Renderer passes the user's preferred starting format; we sort filters so it
      // shows up as the default selection in the OS dialog.
      const xlsxFilter = { name: 'Excel Workbook', extensions: ['xlsx'] }
      const csvFilter = { name: 'CSV', extensions: ['csv'] }
      const filters =
        format === 'csv' ? [csvFilter, xlsxFilter] : [xlsxFilter, csvFilter]
      const result = await dialog.showSaveDialog(win, {
        title: 'Export BOQ',
        defaultPath,
        filters
      })
      if (result.canceled || !result.filePath) return null
      // Detect chosen format from the filename's extension (Pitfall 7 — no filterIndex).
      const lowered = result.filePath.toLowerCase()
      if (lowered.endsWith('.csv')) {
        return { filePath: enforceCsvExtension(result.filePath), format: 'csv' }
      }
      // Default to xlsx (handles .xlsx and any other / no extension).
      return { filePath: enforceXlsxExtension(result.filePath), format: 'xlsx' }
    }
  )

  // Atomic XLSX write: build buffer via ExcelJS, write to .tmp + rename.
  // Wraps in try/catch + discriminated union per D-21 (Phase 4.1 pattern mirror).
  ipcMain.handle(
    'file:writeBoqXlsx',
    async (
      _event,
      filePath: string,
      structure: BoqStructure
    ): Promise<{ ok: true } | { ok: false; reason: string }> => {
      try {
        const buf = await buildBoqXlsx(structure)
        await atomicWriteFile(filePath, buf)
        return { ok: true }
      } catch (err) {
        const reason = err instanceof Error ? err.message : String(err)
        return { ok: false, reason }
      }
    }
  )

  // Atomic CSV write: build text via csv-stringify, wrap in UTF-8 Buffer, atomic write.
  ipcMain.handle(
    'file:writeBoqCsv',
    async (
      _event,
      filePath: string,
      structure: BoqStructure
    ): Promise<{ ok: true } | { ok: false; reason: string }> => {
      try {
        const csvText = buildBoqCsv(structure)
        await atomicWriteFile(filePath, Buffer.from(csvText, 'utf-8'))
        return { ok: true }
      } catch (err) {
        const reason = err instanceof Error ? err.message : String(err)
        return { ok: false, reason }
      }
    }
  )
```

NOTE on the `format` parameter to `dialog:saveExport`: it's the renderer's HINT for which filter to show first. The renderer passes the user's last-used format (or `'xlsx'` as a sensible default — D-16 says XLSX is the primary). The handler uses the hint ONLY to reorder the filters list; the actual returned format is determined by the chosen file's extension.
