<objective>
Remove the `optimizeDeps.exclude: ['pdfjs-dist']` entry from the renderer block in `electron.vite.config.ts` so Vite's esbuild optimizer pre-bundles pdfjs-dist into a single file. The current exclusion forces the dev server to serve pdfjs-dist v5 as raw ESM, triggering hundreds of individual HTTP requests to `localhost:5173` for each internal import. The renderer never finishes loading, `ready-to-show` never fires, and the window stays hidden because of `show: false`.

Purpose: The exclusion was originally added in Phase 01 to fix a worker URL issue, but the actual fix was the `?url` import on line 2 of `pdf-setup.ts` (`import rawWorkerUrl from 'pdfjs-dist/build/pdf.worker.mjs?url'`). Vite's `?url` asset handling is independent of dep optimization — it works the same whether pdfjs-dist is pre-bundled or not. Removing the exclude restores fast renderer startup without breaking the worker loader.

Output: Patched `electron.vite.config.ts` with the `optimizeDeps` block removed from the renderer config; dev server starts, app window appears, PDF rendering continues to work.

<background>
**Root cause** (from upstream diagnosis):
- `pdf-setup.ts` does `import * as pdfjsLib from 'pdfjs-dist'` at module load.
- With `optimizeDeps.exclude: ['pdfjs-dist']`, Vite skips pre-bundling and serves pdfjs-dist as raw ESM.
- pdfjs-dist v5.5.207 has hundreds of internal ESM imports → hundreds of individual HTTP requests on dev startup.
- The renderer never finishes loading → `ready-to-show` never fires → BrowserWindow stays hidden (`show: false`).

**Why removing the exclude is safe:**
- The `?url` import on line 2 of `pdf-setup.ts` uses Vite's asset-URL handling, which is entirely separate from dep optimization. It will continue to resolve `pdf.worker.mjs` to a fingerprinted asset path regardless.
- The blob-wrapper worker pattern (lines 15-83) and `GlobalWorkerOptions.workerSrc = blobWorkerUrl` (line 142) are unaffected by what Vite does with the main `pdfjs-dist` import.
- pdfjs-dist v5.5.207 is self-contained ESM with no external deps, so esbuild's optimizer will bundle it cleanly.
- `viteStaticCopy` of `pdf.worker.mjs` to the renderer output dir (lines 26-32) is independent of optimizeDeps and remains in place for production builds.

**Prior history:** Quick task `260429-jov` (commit 0442ebc, completed earlier today) RESTORED this exclude based on a faulty hypothesis that it was needed for the worker URL. That hypothesis was wrong — this task reverses that decision with the correct diagnosis.
</background>
